[폴더 구조] App Router 실무 표준과 코로케이션(Co-location) 전략

📅 2026-06-27 · 섹션 5 코로케이션App Router PPRPrivate 폴더모노레포
📁 App Router는 page.tsx·route.ts만 라우팅으로 인정해 코로케이션(관련 파일을 라우트 폴더에 응집)을 가능케 했다. src/ 기반 표준 트리(app·components·hooks·lib·utils·types·actions)로 뼈대를 세우되, 전역 공통만 루트에, 특정 페이지 전용은 그 라우트 폴더·_private 폴더에 둔다. 핵심 분별: lib(외부 의존·부수효과) vs utils(순수 함수), _폴더(라우팅 제외). 규모가 더 커지면 모노레포로 점진 확장.

1. 맥락 — 상태(혈관)를 잡았으니 이제 골격(구조)

섹션 4까지 Server Actions·Zustand로 데이터(상태)를 통제했다. 이제 그 혈관들이 엉키지 않게 튼튼한 골격(폴더 구조)을 세운다.

당위성 폴더 구조는 대규모 실무에서 유지보수의 80%를 결정. "새로 합류한 동료가 폴더 트리만 보고도 앱의 기능·데이터 흐름을 유추할 수 있다면" 그것만으로 좋은 설계다.

2. 패러다임 변화 — Pages Router vs App Router (코로케이션)

📦 Pages Router (과거)

  • pages/ 안 모든 파일이 곧 URL
  • 전용 컴포넌트·스타일을 라우팅 폴더 밖 멀리 둬야 함
  • 관련 파일이 흩어짐

🍽️ App Router (현재) — 코로케이션

  • page.tsx·route.ts만 라우팅으로 인정
  • 그 외 파일은 같은 라우트 폴더에 응집 가능
  • 전용 컴포넌트·테스트·스타일을 한곳에
🧊
과거 = 모든 식재료를 거대한 공용 냉장고(루트)에 몰아두고 매번 뒤적 · 현재 = 스테이크 전용 냉장고 / 파스타 전용 조리대처럼 역할별로 재료·도구를 한곳에 응집.

3. 왜 구조가 성능까지 결정하나 — PPR과 Suspense 경계

2026년 Next.js 16에선 PPR(Partial Prerendering, 부분 사전 렌더링)이 안정화되어 실무 기본. 정적인 부분은 미리 렌더, 동적 데이터만 나중에 스트리밍한다.

구조 ↔ 성능 폴더를 잘 나눠 컴포넌트를 분리하면 Next 엔진이 어디를 먼저 보여주고 어디를 기다리게 할지(Suspense Boundary)를 더 영리하게 판단 → 각 컴포넌트가 독립 스트리밍 경계를 갖기 쉬워져 성능 극대화.

4. 실무 표준 폴더 트리 (src/ 기반)

전역 공통 재료들의 근원지는 src/(source). 규모가 커져도 흔들리지 않는 뼈대.

src/
├─ app/          // 앱의 심장 · 라우팅 중심
│  ├─ (auth)/      // Route Group: URL에 안 드러나는 논리적 묶음
│  ├─ _components/ // Private: 라우팅에서 제외된 내부 전용
│  ├─ layout.tsx
│  └─ page.tsx
├─ components/   // 범용 공통 UI (Button/Modal/Input) — "단순"
├─ hooks/        // 커스텀 훅 (useWindowSize, useOutsideClick)
├─ lib/          // 서드파티 설정·인스턴스 (axios/zustand/prisma)
├─ utils/        // 순수 함수 (formatDate, formatCurrency)
├─ types/        // 전역 TS 타입 (User, Product)
└─ actions/      // Server Actions (loginAction, addToCart)
폴더의미 · 무엇을 두나핵심 성격
app/라우팅 중심. Route Group (auth), Private _folder, layout/page탐색 여정 결정
components/앱 전체 재사용 범용 UI(버튼·모달·인풋)Dumb — props로만 그림
hooks/상태·생명주기 낚아채는 커스텀 훅로직 분리·재사용
lib/axios·zustand store·prisma 등 외부 인스턴스외부 의존·부수효과
utils/날짜·통화 포맷 등순수 함수(같은 입력→같은 출력)
types/전역 인터페이스(User·Product)규격 통일·자동완성
actions/로그인·장바구니 등 서버 로직FE↔BE 경계 매끄럽게
components의 "단순"은 칭찬 서버 데이터를 직접 부르거나 비즈니스 로직을 판단하지 않고, 오직 부모가 준 props로 화면만 예쁘게 그린다는 긍정적 의미.
actions 응집 팁 공통 액션은 src/actions/, 특정 기능 전용 액션은 그 페이지 폴더 안에 둬 응집도를 극대화. API 라우트 없이 폼·버튼에서 직접 호출.

5. 실무 분별 3대 포인트 (가장 많이 헷갈리는 곳)

① components 딜레마 — 무지성 루트 수집은 안티패턴

루트 components/엔 디자인 시스템급 범용 공통 UI만. "장바구니 페이지 전용 카트 요약 보드" 같은 건 그 라우트 폴더 안 또는 _components에 둔다.

이유 기능 수정·삭제 시 흩어진 파일을 찾을 필요 없이 그 폴더만 관리하면 끝.

② lib vs utils — 외부 의존 여부로 가른다

🧮 utils — 순수 로직

  • 외부 의존 X, 부수효과 X
  • 같은 입력 → 항상 같은 출력
  • 테스트 쉬움 · 서버/클라 어디서든 안전
  • 예: formatDate, formatCurrency

🔌 lib — 외부 브릿지

  • 외부 시스템/라이브러리에 의존
  • 무겁고 부수효과 있는 설정
  • 외부 생태계 ↔ 우리 앱 연결
  • 예: axios 설정, zustand store, prisma
유지보수 이점 외부 의존 설정을 lib/에 중앙집중 → 라이브러리 교체·버전업 시 이 폴더만 수정.

③ Private 폴더 _folderName — 라우팅 은닉화

폴더명 앞에 언더스코어(_)를 붙이면 Next 라우터가 투명인간 취급해 라우팅에서 완전히 제외.

  • 실수로 page.tsx를 만들어도 원치 않는 경로 노출 사고를 원천 봉쇄(보안)
  • 대시보드 전용 차트·위젯을 _components로 묶으면 글로벌 오염 없이 도메인 내 응집
→ 이는 마이크로서비스 / 기능 분할(feature-sliced) 설계 철학과 맞닿은 모던 프론트엔드 핵심 기법.

6. 더 커지면 — 모노레포로 점진 확장

기업 규모로 거대해지면 단일 앱을 넘어 모노레포(단일 저장소) 고려. 관리자 웹·사용자 웹·공통 UI 라이브러리를 한 GitHub 저장소에서 관리.

✅ 장점

  • 공용 컴포넌트·유틸 여러 앱에서 공유
  • 패키지 의존성 일괄 관리
  • 대규모 리팩토링 유리

⚠️ 단점

  • Turborepo·Yarn Workspaces 등 초기 구축 까다로움
  • 학습 곡선 가파름
  • 빌드 속도·배포 파이프라인 부담 ↑
✅ 실무 전략 처음부터 무리하지 말고 단일 앱 표준 구조로 시작 → 팀 규모·복잡도에 따라 모노레포로 점진 확장.
✅ 핵심 정리
· App Router = page/route만 라우팅 → 코로케이션으로 관련 파일 응집
· 표준 트리: app·components·hooks·lib·utils·types·actions (src 기반)
· 전역 공통만 루트, 전용은 라우트 폴더·_private로
· lib=외부 의존, utils=순수 · _폴더=라우팅 제외(보안·응집)
· 단일 앱 → 필요 시 모노레포 점진 확장

23강. 코드의 질서를 만드는 마법

[생산성] 절대 경로 설정(@)과 케밥-파스칼-카멜 케이스 네이밍 컨벤션

📅 2026-06-27 · 섹션 5 절대 경로tsconfig paths Subpath Imports네이밍 컨벤션
🧭 끝없는 상대 경로(../../../)는 가독성·유지보수의 악몽. tsconfig.json의 baseUrl+paths로 @/* → src/* 절대 경로를 깐다(우편번호 부여). 네이밍은 폴더=kebab-case(URL·OS 안전), 컴포넌트=PascalCase(JS/HTML과 구분), 훅·유틸=camelCase(순수 로직 표시). 마지막 핵심 팁: index.ts 남용 금지 — 파일명을 명확히 써 검색·식별성을 높인다.

1. 맥락 — 뼈대(22강)를 세웠으니 길(경로)·이름(컨벤션)을 정비

22강에서 폴더 뼈대를 세웠다면, 이번엔 그 폴더들을 코드에서 불러올 때의 경로와 이름 짓는 규칙을 정한다. 둘 다 "팀원이 같은 문법으로 소통하기 위한 약속".

2. 상대 경로의 두 가지 악몽

❌ ../../../ 의 문제

  • 가독성 저하 — 점·슬래시가 이어져 경로 파악이 힘듦
  • 유지보수 악몽 — 파일을 다른 폴더로 옮기면 연결된 경로가 전부 깨짐

3. 해결 — tsconfig.json 절대 경로 (단 두 줄)

프로젝트 최상단 tsconfig.json의 compilerOptions에 baseUrl(기준점)과 paths(별칭 매핑)를 추가.

// tsconfig.json
{
  "compilerOptions": {
    "baseUrl": ".",            // 루트를 기준점으로
    "paths": {
      "@/*": ["src/*"]       // @ → src 하위 전체
    }
  }
}

❌ 상대 경로

import { Button } from
  '../../../components/common/Button';

✅ 절대 경로

import { Button } from
  '@/components/common/Button';
📮
복잡한 골목길 주소(상대 경로) 대신 명확한 우편번호(@) 부여 → 어느 주방장이든 파일 위치를 한눈에 파악하고 즉시 가져온다.

4. [2026 실무 팁] package.json Subpath Imports (#)

Next.js 16에선 tsconfig 외에 package.json의 imports로도 별칭을 줄 수 있다. 샵(#)으로 "내부 모듈"임을 명시.

// package.json
{
  "imports": {
    "#*": "./src/*"
  }
}
// 사용
import { authService } from '#services/auth';
왜 표준이 되어가나 별도 빌드 도구 설정 없이 Node.js 표준을 따르므로, 라이브러리 개발·대규모 엔터프라이즈에서 점차 표준으로 자리잡는 중. (런타임 성능도 미세 최적화)

5. 네이밍 컨벤션 — 대상에 따라 3가지

대상케이스예시이유
폴더kebab-caseuser-profile/URL과 일치 · OS 대소문자 라우팅 오류 차단
컴포넌트 파일PascalCaseUserProfileCard.tsxJS 함수·HTML 태그와 시각적 구분
훅·유틸 파일camelCaseuseWindowSize.ts · formatDate.ts"컴포넌트 아닌 순수 로직"임을 파일명으로 표시
  • 폴더(kebab) — 라우트 그룹 (auth)·프라이빗 _internal-logic도 소문자 기반
  • 컴포넌트(Pascal) — 스타일·테스트를 묶을 땐 폴더명=파일명 선호 (UserProfileCard/UserProfileCard.tsx)
  • 훅·유틸(camel) — 파일명만 보고도 "화면이 아니라 로직"임을 즉시 유추

6. [가장 중요한 팁] index.ts 남용 금지

과거엔 import 경로를 줄이려 폴더마다 index.ts를 뒀지만, 에디터 탭에 index 파일만 잔뜩 열려 "지금 뭘 수정 중인지" 헷갈린다.

❌ index 파일명

import UserProfileCard from
  '@/components/UserProfileCard';
// 실제 파일은 index.tsx → 탭에서 식별 어려움

✅ 명확한 파일명

import UserProfileCard from
  '@/components/UserProfileCard/UserProfileCard';
// 검색·가독성·유지보수 ↑
→ 명확한 컴포넌트 이름을 파일명으로 직접 사용해 검색·유지보수 효율을 극대화.
✅ 핵심 정리
· 절대 경로: tsconfig baseUrl+paths로 @/* → src/* (상대 경로 악몽 제거)
· 2026 팁: package.json Subpath Imports # (Node 표준)
· 네이밍: 폴더=kebab · 컴포넌트=Pascal · 훅/유틸=camel
· index.ts 남용 금지 — 파일명을 명확히
다음 강 예고 (24강) 이 뼈대 위에서 Next.js의 꽃 서버 컴포넌트 vs 클라이언트 컴포넌트를 어떻게 효율적으로 분리·배치할지.

24강. 서버와 클라이언트의 공존

[컴포넌트 설계] 뿌리(Server)·열매(Client) 전략과 합성(Composition) 패턴

📅 2026-06-27 · 섹션 5 서버 컴포넌트Thin Client 합성 패턴server-onlyuse cache
🌳 App Router에선 기본이 서버 컴포넌트(뿌리·줄기) — 데이터·SEO·보안이 필요하면 무조건 여기서 시작. 'use client'는 상호작용이 일어나는 말단(열매)에만 최소화(Thin Client). 클라이언트가 서버를 감싸야 할 땐 직접 import 말고 children 슬롯으로 끼워 넣는 합성 패턴으로 서버 이점을 보존. 민감 로직엔 import 'server-only'로 브라우저 유출을 빌드 단계에서 차단.

1. 대전제 — 기본은 서버 컴포넌트 (뿌리·줄기)

App Router에선 'use client' 선언이 없으면 모든 컴포넌트가 서버 컴포넌트. 땅속에서 데이터를 길어 올리고 뼈대를 세우는 역할.

🌳 서버 컴포넌트가 좋은 이유

  • 속도 — 서버 내부망에서 초고속으로 데이터를 가져와 완성된 HTML을 던짐(브라우저 로딩 대기 X)
  • PPR — 정적 요소는 즉시, 동적 데이터만 나중에 채워 체감 성능 ↑
  • 보안 — 민감 API 접근 로직이 브라우저에 노출 안 됨
  • 무게 — 무거운 이미지·계산이 서버에서 처리되어 전송
🔥 제1원칙 데이터가 필요하거나 SEO가 중요하면 → 무조건 서버(뿌리)에서 시작.

2. 클라이언트는 "열매"로 최소화 — Thin Client

버튼 클릭·타이핑 등 상호작용 순간에만 'use client'. 핵심은 클라이언트 경계를 나무 끝 열매로 밀어내기.

❌ Fat Client

'use client';

Q. 24강의 핵심 주장(기본 서버 컴포넌트·Thin Client·합성 패턴·use cache·server-only)에 틀리거나 과장된 부분은 없나?
🔍 전반적으로 정확하고 실무 정설에 부합. 다만 3곳은 보정 필요: ① 'use cache'는 오랫동안 experimental(dynamicIO)이었고, 예제처럼 fetch의 next:{revalidate}와 섞으면 캐싱 모델이 충돌(use cache는 cacheLife/cacheTag로 제어). ② "PPR 완전 안정화"는 미래 시점 가정 — 실제로 오래 experimental이었다. ③ "감싸면 자식이 클라가 된다"의 진짜 원인은 "감쌈"이 아니라 "클라 파일에서 import"다.

1. "감싸면 클라가 된다" → 정확히는 "클라 파일에서 import하면"

강의 결론(합성 패턴)은 옳지만, 원인을 "감싸서"로 설명하면 오해가 생긴다. 진짜 규칙:

📐 경계 전파 규칙

  • 'use client'가 붙은 모듈이 import하는 모든 컴포넌트는 클라이언트 번들에 들어간다 → 그 컴포넌트는 더 이상 async 서버 패칭·server-only import 불가.
  • 반대로 children/props로 "전달"받은 것은 부모의 클라 여부와 무관 — 호출(렌더)한 곳(부모의 부모=서버)에서 이미 RSC로 렌더돼 결과만 흘러든다.

두 방향은 완전히 비대칭이다

관계결과이유
서버가 클라를 import✅ 정상 (표준 패턴)렌더가 서버→클라로 흐름. 경계에서 "구멍"만 남기고 props 직렬화해 전달

클라가 서버를 {children}/props로 받음✅ 서버 유지 (합성 패턴)import가 아니라 "전달". 상위 서버에서 이미 RSC로 렌더됨

2. 'use cache' 예제 — opt-in 전제 + 두 캐싱 레이어 혼용

🔎 출처 Next.js 공식 use cache 지시어 문서 & "Migrating to Cache Components" 가이드(v16.2.9). 아래는 문서 확인 사실.

⚠️ 무엇을 짚어야 하나

  • 설정 전제(검증됨) — 'use cache'는 Cache Components 기능으로, next.config.ts에 cacheComponents: true를 켜야 동작한다(과거 experimental.dynamicIO/useCache를 Next 16이 이 이름으로 대체). "지시어 한 줄로 끝"이 아니라 opt-in 설정이 선행. Next 16 기본은 모든 동적 코드가 요청 시 실행(캐싱은 전부 opt-in).
  • 제어 일원화 권장 — 'use cache' 함수의 수명은 cacheLife()(미설정 시 default 프로필: stale 5분 / revalidate 15분), 무효화는 cacheTag()로 한다. 공식 마이그레이션 가이드는 fetch의 revalidate·tags를 cacheLife·cacheTag로 옮기라고 명시.
  • "충돌"은 아님 — "중복"이 정확 — 가이드 원문: "기존 fetch 캐싱은 별도 레이어로 계속 동작한다." 따라서 use cache 안에 next:{revalidate}를 둬도 에러는 아니다. 다만 캐시 수명을 두 곳(fetch Data Cache + use cache 스코프)에서 이중으로 거는 셈이라 비관용적이고 의도가 모호하다.
// 강의 예제 (두 레이어 중복 — 에러는 아니지만 비권장)
async function getUser(id) {
  'use cache';
  await fetch(url, { next: { revalidate: 3600 } }); // fetch 레이어와 중복
}

// 권장 — use cache 스코프로 일원화 (cacheComponents:true)
import { cacheLife, cacheTag } from 'next/cache';
async function getUser(id) {
  'use cache';
  cacheLife('hours');   // 수명 ← fetch revalidate 대체
  cacheTag('user');     // 태그 ← fetch tags 대체
  await fetch(url);
}
저장소 차이도 알아둘 것 fetch Data Cache는 배포·서버리스 인스턴스를 넘어 지속되지만, use cache는 기본 in-memory라 인스턴스 종료 시 사라진다(지속하려면 use cache: remote/커스텀 핸들러). 또 Next 16에선 revalidateTag()가 cacheLife 프로필 인자를 요구하도록 바뀜.
→ 개념 학습용으론 OK지만 "이대로 복붙하면 안 됨". 프로젝트의 Next 버전·cacheComponents 설정을 먼저 확인.

3. "PPR 완전 안정화·실무 기본" — 공식 문서로 검증한 결과

🔎 출처 Next.js 16 공식 릴리스 블로그(2025-10-21). 아래는 추정이 아니라 문서 확인 사실.

강의는 "PPR이 완전히 안정화되어 실무에 기본이 되었다"고 단정한다. 검증 결과 "안정화"는 형태가 다르고, "기본"은 틀렸다.

강의 서술실제(검증)
PPR이 오래 실험 단계였다(암묵)맞음 — 2023 도입 후 Next 16 전까지 experimental.ppr 플래그로만 제공
"완전히 안정화됨"부분적 — Next 16에서 experimental.ppr·experimental_ppr는 제거되고, PPR이 Cache Components 모델로 흡수됨("use cache" 중심)
"실무에 기본이 되었다"아님 — Cache Components는 전적으로 opt-in(cacheComponents: true). 기본은 "모든 동적 코드가 요청 시 실행". 게다가 2025-10 출시로 갓 나온 최신 기능

📐 정리

  • 설계 원칙은 유효 — "정적은 즉시, 동적만 스트리밍 → 컴포넌트를 잘게 나눠 Suspense 경계 확보"는 PPR/Cache Components 어느 쪽이든 그대로 통용.
  • "기본값" 서술은 받아들이지 말 것 — 실제론 cacheComponents: true opt-in. 적용 전 Next 버전·설정 확인이 선행.
  • 네이밍 변화 — 과거 experimental.dynamicIO → Next 16에서 cacheComponents로 개명·승격. (18강 PPR 팩트체크와 동일 결.)

4. 사소하지만 보강할 점

  • "서버 내부망에서 초고속" — DB/API가 서버와 가까울 때의 얘기. 외부 서드파티 API를 호출하면 서버에서도 네트워크 지연은 존재(그래도 클라 왕복+번들은 줄어듦).
  • server-only의 짝 — 클라 전용 보호엔 client-only 패키지가 대칭으로 존재(브라우저 전용 모듈을 서버에서 import 시 에러).
✅ 결론 24강의 멘탈 모델(뿌리=서버 / 열매=클라 / 합성 패턴 / server-only)은 정확하고 실무에서 그대로 통용. 보정 포인트는 셋: 경계는 "감쌈"이 아니라 "import"로 갈린다 · use cache는 cacheComponents:true opt-in이고 수명은 cacheLife로 일원화(fetch revalidate 중복은 비권장, 단 충돌·에러는 아님) · PPR은 Next 16에서 Cache Components로 흡수됐으나 "기본"이 아니라 opt-in(공식 문서 검증).

도메인 분할이 핵심 진화 강의가 끝에 언급한 "기능 분할(feature-sliced) 설계 철학"을, 우리는 libs/domain/*(common·auth·contract·realtor 등 ~27개 도메인)로 패키지 레벨에서 구현. 각 도메인이 자체 components/ hooks/ utils/ type.ts를 가진다 → 강의의 응집 원칙을 더 큰 단위로 확장.

3. 강의 표준과 같은 점 ✅

잘 따르고 있는 원칙

  • 코로케이션 — App Router 앱(realtor)에서 page.tsx 옆에 *Client.tsx·loading.tsx를 같이 둠 (예: vacancy/list/VacancyListPageClient.tsx)
  • Route Group — (admin)·(protected)·(require-profile)로 URL에 안 드러나는 논리적 묶음 + 접근 제어 계층화
  • 전역 공통은 공유, 전용은 가까이 — 범용은 libs/shared, 도메인 전용은 libs/domain/*, 페이지 전용은 라우트 폴더에 코로케이션
  • lib vs utils 분리 — 외부 의존(libs/api) vs 순수 함수(libs/shared/utils)가 실제로 갈라져 있음

4. 강의 표준과 다른 점 ⚠️ (옳고 그름이 아니라 선택)

표준과의 갭

  • Server Actions 미사용 — 강의는 actions/로 FE↔BE를 잇지만, 우리는 React Query(useQuery/useMutation) + axios 전통 패턴. App Router 앱조차 클라이언트단에서 API 호출. (섹션4 낙관적 업데이트 판단과 동일 맥락)
  • Private _folder 미사용 — 강의가 강조한 언더스코어 은닉 폴더 컨벤션은 안 씀. 라우트 그룹 ()로 묶을 뿐, "라우팅 제외 전용 폴더"는 공유 패키지(libs/domain)로 대체. → 도메인 전용 컴포넌트를 라우트 안에 둘 땐 _components 도입 검토 여지.
  • 주력 앱은 아직 Pages Router — realtor·admin은 App Router지만 lessee·lessor 등은 Pages Router. 강의의 코로케이션/PPR 이점은 App Router 앱에서만 적용됨.
  • 앱 루트 components/ 잔존 — apps/realtor/components/처럼 코로케이션과 앱-루트 컴포넌트 폴더가 혼용. 정리 시 "전역=libs, 페이지전용=라우트 폴더" 규칙으로 수렴 여지.
✅ 진단 zaritalk-front는 강의의 단일 앱 표준을 "졸업"하고 모노레포 + 도메인 분할로 진화한 형태다. 코로케이션·Route Group·lib/utils 분리 같은 핵심 원칙은 충족하고, 차이점(Server Actions·_folder 미사용, Pages Router 잔존)은 틀린 게 아니라 우리 환경(네이티브 웹뷰·점진 마이그레이션)에 맞춘 선택. 22강의 진짜 교훈("동료가 트리만 보고 흐름을 유추할 수 있나")으로 보면 — libs/domain 분할 덕에 그 기준은 잘 지켜지는 편.

zaritalk-front 경로 별칭·네이밍·배럴 — 강의 23강에 비춰 보기

📅 2026-06-27 실측@zaritalk 별칭배럴 파일네이밍
Q. 23강의 "@/* → src/* 절대 경로 + 네이밍 + index.ts 남용 금지" 기준으로 보면, 우리 프로젝트의 별칭/네이밍/배럴 전략은 어떤가?
🧭 별칭·네이밍은 강의보다 더 엄격·체계적이다. 강의의 단순 @/* 대신 패키지 단위 별칭 @zaritalk/{패키지}로 의존성 방향을 강제한다. 네이밍(폴더 kebab·컴포넌트 Pascal·훅/유틸 camel)도 정확히 일치. 단 하나 정반대: 강의는 index.ts 남용을 말렸지만 우리는 배럴 파일 679개를 의도적으로 쓴다 — 이건 "남용"이 아니라 패키지 공개 API(진입점) 통제라는 다른 목적의 선택.

1. 경로 별칭 — @/*가 아니라 @zaritalk/{패키지}

강의는 단일 앱 기준 @/* → src/*. 우리는 모노레포라 tsconfig.base.json의 paths에 패키지 단위 별칭을 깐다.

// tsconfig.base.json (발췌)
"@zaritalk/api":            ["libs/api/src/index.ts"],
"@zaritalk/constants":      ["libs/constants/src/index.ts"],
"@zaritalk/domain/common":  ["libs/domain/common/src/index.ts"],
"@zaritalk/hooks/*":        ["libs/shared/hooks/src/*"],
"@zaritalk/ui-components/*": ["libs/shared/ui-components/src/*"],
"@zaritalk/utils/*":        ["libs/shared/utils/src/*"]
  • 패키지 진입점 매핑(.../src/index.ts) + 와일드카드(/*로 내부 깊은 파일 직접 접근) 혼용
  • 도메인 간/공유는 별칭 강제, 도메인 내부는 상대경로 — realtor 앱 실측 별칭 import 164 vs 상대경로 3. 의존성 방향을 별칭으로 통제.
  • Subpath Imports(#)는 미사용 — 강의의 2026 팁(package.json imports)은 우리 프로젝트엔 없음. tsconfig paths만 사용.
→ 강의의 "우편번호" 비유를 패키지 레벨로 확장한 형태. @/가 "이 앱 안"이라면 @zaritalk/는 "이 회사의 어느 패키지".

2. 네이밍 컨벤션 — 강의와 정확히 일치 ✅

대상강의 권장우리 실제예시
폴더kebab-case일치anonymous-talk, real-estate-map, ui-components
라우트 폴더괄호+kebab일치(admin), (protected), (require-profile)
컴포넌트 파일PascalCase일치MainIntro.tsx, BottomNav.tsx, PhoneNumAuthPopup.tsx
훅·유틸 파일camelCase일치useDebounce.ts, useClickOutside.ts, formatAreaAmount.ts
차이 1곳 강의는 컴포넌트를 "폴더명=파일명" 구조(UserProfileCard/UserProfileCard.tsx)로 권하지만, 우리는 대체로 평면 .tsx 파일(기능별 서브폴더로 묶음). 일부 폴더(kakaoSync)만 camelCase로 컨벤션에서 살짝 벗어남.

3. 핵심 충돌 — index.ts 배럴 파일 679개 (강의는 "지양")

📕 강의 입장 — 남용 금지

  • 에디터 탭이 index로 도배 → 식별 어려움
  • 파일명을 명확히 직접 import 권장

📗 우리 입장 — 공개 API 통제

  • 패키지 진입점(src/index.ts)을 단일화
  • 외부엔 index가 export한 것만 공개
  • 내부 파일 구조 리팩토링이 소비자에게 안 새어나감
// 명시적 index import (흔함)
import { Button, Layout } from '@zaritalk/ui-components/index';
// 암시적 index (paths가 자동 매핑)
import { AlertLayer } from '@zaritalk/domain/common';
// index 우회 — 깊은 파일 직접 접근(와일드카드 별칭)
import { useSearchParams } from '@zaritalk/hooks/lib/navigation';
트레이드오프 인지 배럴은 트리쉐이킹 약화·순환참조·초기 번들/빌드 부담 위험이 있다. 그래서 우리도 무거운 깊은 모듈은 @zaritalk/hooks/lib/navigation처럼 index를 우회해 직접 import하는 절충을 이미 쓴다. → 강의의 경고는 "앱 내부 컴포넌트 배럴"에 유효하고, 우리의 배럴은 "패키지 경계 API"라 목적이 다르다.
✅ 진단 별칭·네이밍은 강의 기준을 넘어서 더 체계적(패키지 별칭으로 의존성 방향까지 통제). index.ts만 강의와 정반대로 적극 사용하는데, 이는 단일 앱 관점의 "남용"이 아니라 모노레포 패키지의 공개 API 경계 설정이라는 의도된 선택. 개선 여지는 배럴 우회 규칙 명문화(무거운 모듈은 깊은 경로 직접 import)와 예외 폴더(kakaoSync 등) kebab 정리 정도.
