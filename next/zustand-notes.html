15강. 왜 Zustand인가?

[철학] Context API의 확성기 효과와 Redux의 관료주의를 넘어서

📅 2026-06-21 · 섹션 4 Context APIRedux 클라이언트 상태selectorSSR
🐻 서버 상태는 TanStack Query가, 가벼운 클라이언트 상태(다크모드·토글·장바구니)는 Zustand가 맡는다. Context의 확성기 효과(불필요 리렌더)와 Redux의 관료주의(action·dispatch·reducer)를 모두 걷어내고, selector로 필요한 값만 구독해 해당 컴포넌트의 어깨만 톡톡 친다. Next.js에선 Provider 껍데기 없이 RSC 철학까지 지킨다.

1. 맥락 — 서버 상태에서 클라이언트 상태로

지금까지 TanStack Query로 다룬 건 외부 서버에서 재료를 공수하는 서버 상태. 이번부터는 브라우저 안에서 즉각 반응해야 하는 클라이언트 상태를 다룬다.

🍽️
TanStack Query = 외부에서 재료를 들여오는 주방 시스템(서버 상태) · Zustand = 매장 조명·장바구니처럼 매장 안에서 즉각 일어나는 일을 관리하는 지배인(클라이언트 상태).
  • 예: 다크모드 on/off · 모바일 사이드바 토글 · 장바구니 담은 상품 목록
  • 서버까지 안 가도 되는, 브라우저 내부에서 빠르고 가볍게 반응할 데이터
비율 앱 데이터의 약 80%는 서버 상태(TanStack Query), 20%가 UI 상태(Zustand). 이 분리가 이후 모든 선택의 근거.

2. Context API의 문제 — 'use client' 껍데기 & 확성기 효과

장점: 외부 라이브러리 설치가 필요 없다. 그러나 두 가지 비용이 크다.

  • ① 루트에 'use client' 강제 — Context를 쓰려면 최상단 레이아웃을 useState 가진 Provider로 감싸야 하고, 그러면 루트에 'use client'가 붙는다. RSC 이점을 극대화해야 하는 Next.js 13+에서 구조적으로 우아하지 못하고 관리 포인트를 늘린다.
  • ② 확성기 효과 — 값 하나만 바뀌어도 구독한 모든 컴포넌트가 리렌더된다.
📢
isDark 변경 → Context가 매장 전체에 "조명 꺼졌습니다!" 확성기로 외침 → 조명과 무관한 냉장고·화분·냅킨까지 모두 멈추고 다시 그림. 매장이 커질수록 성능 낭비.

3. Redux의 관료주의 — action · dispatch · reducer

견고하지만, 스위치 하나 켜는 데도 거대한 절차가 필요하다.

📋
전등 하나 켜는데 서류(Action) → 결재(Dispatch) → 장부(Reducer) 기록. 안전하지만 작은 일에도 절차가 과해 개발자를 지치게 한다.
  • 과거엔 서버 데이터+UI 상태를 Redux 하나로 관리 → 무거움을 감수할 이유가 있었음
  • 지금은 서버 상태(80%)를 TanStack Query가 전담 → 남은 20% UI 상태에 Redux는 "압정 하나에 굴착기"
Redux Toolkit은? 보일러플레이트는 줄었지만 store 설정 + Provider로 앱 감싸기라는 관료주의의 본질은 그대로.

4. Zustand의 해법 — 어깨 톡톡 & Provider 불필요

📢 Context — 확성기

  • 루트에 'use client' Provider 강제
  • 값 1개 변경 → 구독 전체 리렌더
  • 무관한 컴포넌트까지 다시 그림

👉 Zustand — 어깨 톡톡

  • 앱 전체 Provider 껍데기 불필요
  • selector로 isDark만 구독 → 그 버튼만 갱신
  • 나머지는 서버 컴포넌트로 남음

Zustand(독일어 "상태")는 구독이 필요한 작은 컴포넌트 최상단에만 'use client'를 붙이면 된다.

나머지 거대한 페이지·레이아웃은 모두 SEO 최적화되고 렌더링 빠른 순수 서버 컴포넌트로 남는다.
→ 철학: 복잡한 설정·무거운 절차를 걷어내 개발자가 문제 해결에만 집중하게 한다.

5. 기본 문법 — create · set · selector · 자동 병합

import { create } from 'zustand';

interface CounterState {
  count: number;
  increment: () => void;
}

export const useCounterStore = create<CounterState>((set) => ({
  count: 0,                                // 초기 상태
  increment: () => set((state) => ({ count: state.count + 1 })),
}));
  • create — 저장소를 만드는 기계(데이터 상자의 출발점)
  • interface CounterState — 상자에 무엇이 들어갈지 적는 설계도
  • set — 값을 바꾸는 유일한 전용 리모컨. 이걸 통해야 React가 리렌더를 눈치챈다
자동 병합 useState와 달리 ...state 복사 없이 바꿀 값만 던지면 Zustand가 나머지와 합쳐준다. 
const count = useCounterStore((state) => state.count);
//                              ^^^^^^^^^^^^^^^^^^^^ selector — 필요한 값만 집어옴

창고에서 필요한 count만 집어오므로, 다른 값이 바뀌어도 이 컴포넌트는 조용히 자기 일만 한다 → 확성기 효과 차단.

6. 서버-클라이언트 결합 & 상태 누수 — createStore + Provider

⚠️ 상태 누수(state bleeding) Next.js 서버는 24시간 켜진 공용 주방. 전역 변수로 상태를 두면 여러 사용자가 한 프로세스를 공유해 A 손님의 데이터가 B 손님 요리에 섞인다. → 손님마다 개인용 소금통이 필요. 
// store/counterStore.ts
import { createStore } from 'zustand/vanilla';

export const createCounterStore = (initCount: number) => {
  return createStore<CounterState>((set) => ({
    count: initCount,
    increment: () => set((s) => ({ count: s.count + 1 })),
  }));
};
  • createStore(zustand/vanilla) — React와 분리된 순수 데이터 상자. 서버는 화면을 그리는 곳이 아니므로 더 가볍고 독립적인 이 도구를 쓴다.
  • createCounterStore(initCount) — 상자가 아니라 "상자를 만드는 공장". 접속할 때마다 실행해 그 손님만의 깨끗한 상자를 만든다(체크인마다 새 비누).

데이터 여정 (서버 → 화면)

  1. 서버 컴포넌트가 DB에서 사용자 정보를 가져옴
  2. 그 정보를 공장 함수에 넣어 전용 store 인스턴스 생성
  3. 인스턴스(명함)를 Provider 우산에 담아 앱에 뿌림
  4. 클라이언트는 재요청 없이 서버가 채운 초기 상태를 그대로 물려받아 빠르게 렌더
Provider 성능 걱정은? Provider엔 변하는 값이 아니라 고정된 store 인스턴스만 담는다(value={storeRef.current}). 전화번호(명함)는 값이 바뀌어도 안 변하므로 하위를 줄줄이 리렌더시키지 않는다. 각 컴포넌트는 자기 selector로만 구독.
✅ 핵심 정리
· 서버 상태=TanStack Query, 클라이언트 상태=Zustand · Context=확성기(전체 리렌더), Redux=관료주의(절차 과다) · Zustand=selector로 필요한 값만 구독(어깨 톡톡) + Provider 껍데기 불필요 · 서버에선 요청마다 store 인스턴스를 새로 찍어 상태 누수 방지(Provider엔 인스턴스만)
다음 강 예고 (16강) ① 서버에서 가져온 초기 장바구니 데이터를 store에 미리 채우기 ② localStorage 저장 시 발생하는 Hydration Mismatch를 서버 렌더링 지원 없이 우아하게 해결하기.

16강. 렌더링 지연 없는 Store 구성

[서버 오염 방지] 싱글톤의 위험성과 App Router 전용 Provider 패턴

📅 2026-06-21 · 섹션 4 상태 누수싱글톤 Provider 패턴SSR
🐻 "Zustand는 Provider 불필요"는 SPA 한정. App Router(서버+SSR)에선 전역 싱글톤 store가 모든 사용자에게 공유되어 상태 누수(보안 사고)를 낸다. 해법은 요청마다 store 인스턴스를 새로 찍어 Provider로 전달. 단 Provider엔 변하는 값이 아니라 store 인스턴스(명함)만 담아 폭포수 렌더링을 피한다.

0. 15강과의 "모순" 짚기

15강: "Zustand는 앱 감싸는 껍데기가 필요 없다" → 16강: "Provider 패턴을 배운다". 모순처럼 보이지만 환경이 다르다.

핵심 SPA에선 Provider 불필요가 맞고, Next.js 서버 환경에선 누수 방지를 위해 Provider를 다시 쓴다. 단 Context와 달리 "담는 내용"이 다르다(아래 4번 — Provider엔 변하는 값이 아니라 store 인스턴스만).

1. 기초 문법 복습 — create · set · 함수형 업데이트

import { create } from 'zustand';

interface CounterState {
  count: number;
  increment: () => void;
  decrement: () => void;
}

export const useCounterStore = create<CounterState>((set) => ({
  count: 0,
  increment: () => set((state) => ({ count: state.count + 1 })),
  decrement: () => set((state) => ({ count: state.count - 1 })),
}));
  • 상태 + 액션이 한 집에 — 데이터(count)와 그걸 바꾸는 행동(increment)을 한 객체에 묶는다.
  • set이 유일한 변경 도구 — 직접 값을 꽂으면 React가 변경을 모르고 화면을 안 그린다. 반드시 set을 통해야 리렌더 신호가 간다.
  • 함수형 업데이트 set((state) => ...) — 매개변수 state는 항상 최신값. 버튼을 다다닥 눌러도 이전 값을 안전히 가져와 +1. (useState 함수형 업데이트와 동일 원리)
  • 자동 병합 — count만 던져도 name·age 등 나머지는 그대로 유지. ...state 복사 불필요.

2. 폭포수(useState/Context) vs 어깨 톡톡(Zustand)

🌊 폭포수 렌더링

  • 최상위 useState → props/context로 전달
  • 상태 하나 변경 → 부모+모든 자식 줄줄이 리렌더
  • "위에서 물 부으면 아래가 다 젖음"

👉 어깨 톡톡 렌더링

  • selector로 필요한 값만 콕 구독
  • 구독한 count가 그대로면 다른 게 수백 번 바뀌어도 미동 없음
  • 교실에서 필요한 학생 어깨만 톡톡

SPA라면 create + hook 하나로 상태 관리가 끝난다. 문제는 다음.

3. 왜 전역 싱글톤 store가 서버에서 위험한가 — 상태 누수

⚠️ 상태 오염 시나리오 Next.js 서버는 24시간 켜진 공용 주방(단일 프로세스). 전역 store를 하나만 두면 → A 손님이 카운터를 1로 올림 → 공용 메모리가 1로 변경 → B 손님 화면에도 1이 노출. 남의 장바구니·개인정보가 보이는 대형 보안 사고.

브라우저는 사용자 PC마다 store가 독립 생성돼 문제없다. 위험한 건 서버 쪽 — 여러 요청이 한 메모리를 공유하기 때문.

4. 해법 — 요청마다 store 인스턴스 + Provider (명함만 전달)

손님이 접속할 때마다 그 손님 전용 store를 새로 찍어 쥐여준다. 이때 Provider는 "전용 직원의 명함(전화번호)"만 배급한다.

📇
확성기로 "카운터 1로 바뀜!" 외치는 게 아니라(=Context), "A손님 담당 직원 번호는 010-1234-5678"이라고 명함만 한 번 돌리고 끝. 명함은 상태가 변해도 안 바뀌므로 하위가 줄줄이 리렌더되지 않는다. 자식은 명함을 보고 직원에게 직접 연락해 selector로 필요한 값만 받는다.
// 1) 사용자별 독립 store 공장 (zustand/vanilla)
export const createCounterStore = (initCount: number) => {
  return createStore<CounterState>((set) => ({
    count: initCount,                                  // 서버가 채운 초기값
    increment: () => set((s) => ({ count: s.count + 1 })),
  }));
};

// 2) Provider 우산 — value엔 '명함(인스턴스)'만
export function CounterProvider({ children, initCount }) {
  const storeRef = useRef();
  if (!storeRef.current) {
    storeRef.current = createCounterStore(initCount); // 1회만 생성
  }
  return (
    <CounterContext.Provider value={storeRef.current}>
      {children}
    </CounterContext.Provider>
  );
}

// 3) 자식: 명함으로 직원에게 연락해 필요한 값만 구독
const count = useCounter((state) => state.count);
  • createCounterStore(initCount) — 상자가 아니라 "상자 공장". 접속마다 깨끗한 상자를 찍고 initCount(서버 조사 초기 데이터)를 미리 채운다.
  • useRef + if (!storeRef.current) — 리렌더돼도 새 상자를 또 만들지 않게 처음 그 상자를 붙잡는 끈.
  • value={storeRef.current} — 변하는 숫자가 아니라 전화번호(명함)만 담음 → 폭포수 차단.

5. 완성된 흐름 (서버 → 화면, 지연 0)

  1. 서버가 접속마다 사용자 전용 store 인스턴스를 새로 생성 → 상태 섞임 원천 차단
  2. 서버 컴포넌트가 DB를 먼저 조회해 초기 데이터(예: 장바구니) 확보
  3. 그 데이터를 store(전담 직원)에 쥐여주고 명함을 Provider로 배포
  4. 클라이언트는 재요청 없이 서버가 채운 초기 상태를 그대로 물려받아 빠르고 부드럽게 렌더
✅ 아키텍처의 꽃 Next.js의 서버 데이터 패칭 + Zustand의 섬세한 클라 상태 관리를 지연 시간 없이 결합. Provider를 쓰되 명함(인스턴스)만 넘겨 Context의 폭포수는 피한다.

6. 동작 검증 체크리스트 (강의 노트)

  • 초기 데이터 전달 — 접속 즉시 로딩 없이 서버 초기값(예: 10)이 바로 보이는가
  • 오염 차단 — 창 2개 중 한쪽에서 숫자를 올려도, 다른 쪽 새로고침 시 초기값 그대로인가
  • 렌더 성능 — React DevTools 하이라이트로 숫자만 반짝이고 로고·메뉴바는 가만히 있는가
  • 하이드레이션 일치 — 새로고침 시 0 → 10으로 튀는 깜빡임이 없는가
다음 강 예고 (17강) 일반 create 대신 createStore(zustand/vanilla) + React 19 최신 Context 문법으로 이 "명함표"를 직접 구현.

17강. 안전한 스토어 공장 세우기

[실전 구현] createStore 바닐라 모드와 React 19 use 훅의 만남

📅 2026-06-21 · 섹션 4 createStorevanilla use 훅React 19SSR
🏭 16강 이론(요청별 store + 명함만 전달)을 코드 3단계로 구현. ① createStore(zustand/vanilla)로 React와 분리된 store 공장 → ② 'use client' Provider에서 useState 지연 초기화로 1회만 생성(React 19 동시성 안전, useRef보다 권장) → ③ React 19 use 훅으로 Context를 꺼내는 전용 훅. 마지막에 persist의 hydration 불일치 문제 예고.

왜 create가 아니라 createStore(vanilla)인가

🧷 create (기본) — 서버에서 위험

  • 만들어지는 순간 React와 결합된 전역 변수

📓 createStore (vanilla) — 공장 가능

  • React 사이클과 독립된 순수 JS store 객체
  • 훅에 고정 안 됨 → 요청마다 새로 찍기 유리
  • 접속자별 깨끗한 인스턴스(명함) 생성
📓
create = 벽에 고정된 게시판(공용) · createStore = 가방에 쏙 들어가는 개인 수첩. 손님마다 새 수첩을 꺼내주기 좋다.

1단계 — store 공장 (createStore factory)

// src/store/cartStore.ts
import { createStore } from 'zustand/vanilla';

export interface CartState { items: number; add: () => void; }

export const createCartStore = (initState: number = 0) => {
  return createStore<CartState>()((set) => ({
    items: initState,                                  // 서버 초기값 주입구
    add: () => set((s) => ({ items: s.items + 1 })),
  }));
};
  • 공장 함수 — 상자가 아니라 "상자 만드는 공장". 호출마다 새 메모리에 독립 store 생성.
  • initState — 서버가 미리 조사한 초기 데이터를 채우는 입구(기본 0).
  • createStore<T>()(...)의 빈 괄호 () — TS 추론을 위한 curried 시그니처(타입만 주고 구현은 다음 괄호).

2단계 — Provider (명함 배급소)

'use client';
import { createContext, useState } from 'react';
import { createCartStore } from '../store/cartStore';

export const CartContext = createContext<ReturnType<typeof createCartStore> | null>(null);

export function CartProvider({ children, initialItems = 0 }) {
  const [store] = useState(() => createCartStore(initialItems)); // 지연 초기화: 1회만
  return <CartContext value={store}>{children}</CartContext>;       // React 19: .Provider 생략
}
'use client'의 진짜 의미 "이 컴포넌트부터 클라에서만 렌더"가 아니라 "브라우저 전용 API가 있으니 클라 JS 번들에 포함"하라는 뜻. children을 prop으로 받는 패턴이라, Provider가 클라여도 children은 여전히 서버에서 HTML로 렌더되어 내려온다. (우산은 클라, 내용물은 서버)
⭐ 핵심 꿀팁 — useState 지연 초기화 과거엔 useRef로 store를 담았지만, React 19 동시성 렌더링에선 렌더가 중단·재실행될 수 있다. useState(() => createCartStore(...))처럼 콜백을 주면 최초 1회만 안전하게 인스턴스를 만들어 더 완벽한 방어책. (useRef 패턴 대체)
React 19 Context 문법 <CartContext.Provider value={...}> 대신 <CartContext value={...}>로 .Provider 생략 가능.

3단계 — 전용 훅 (React 19 use)

import { use } from 'react';
import { useStore } from 'zustand';

export function useCartStore<T>(selector: (s: CartState) => T): T {
  const store = use(CartContext);                // useContext보다 간결
  if (!store) throw new Error('CartProvider 안에서 사용하세요');
  return useStore(store, selector);             // vanilla store ↔ React 연결 + selector 구독
}
  • use(CartContext) — React 19 신규 훅. useContext를 대체해 명함(store 인스턴스)을 꺼냄.
  • null 가드 — Provider 우산 밖에서 쓰면 에러로 방어.
  • useStore(store, selector) — 바닐라 store를 React에 연결하고, 16강의 selector "어깨 톡톡"이 그대로 작동.

18강. 새로고침에도 끄떡없는 데이터

[지속성] Persist 미들웨어와 하이드레이션 불일치 완벽 해결

📅 2026-06-21 · 섹션 4 persistskipHydration rehydrate하이드레이션
🔒 persist 미들웨어로 store를 localStorage에 저장 → 새로고침에도 생존. 하지만 서버 HTML(0개) ≠ localStorage(5개) → 하이드레이션 불일치 에러. 해법: skipHydration: true로 자동 주입을 멈추고, Provider의 useEffect에서 persist.rehydrate()를 안전한 타이밍에 수동 호출. 남은 0→5 깜빡임은 isMounted 전략으로 차단.

0. Persist란 — Cache와 헷갈리지 말 것

persist(지속되다) = 휘발성 RAM의 상태를 localStorage(영구 창고)에 저장해 새로고침·재부팅에도 살아남게.

🚀 Cache (TanStack Query)

  • 목적: 속도 최적화(빨리 가져오기)
  • 유통기한 지나면 사라짐

🔒 Persist (Zustand)

  • 목적: 생존(오래 보존)
  • 사용자가 비우기 전까지 영구

1. persist 미들웨어 적용 (store 공장)

import { persist, createJSONStorage } from 'zustand/middleware';

export const createCartStore = (initState = 0) =>
  createStore<CartState>()(
    persist(
      (set) => ({ items: initState, add: () => set((s) => ({ items: s.items + 1 })) }),
      {
        name: 'cart-storage',                      // localStorage 키(이름표)
        storage: createJSONStorage(() => localStorage),
        skipHydration: true,                     // ⭐ 자동 주입 OFF
      }
    )
  );
  • persist(레시피, 옵션) — store 로직을 감싸 변경 시마다 자동 저장.
  • name — 창고에서 우리 데이터를 찾는 키.
  • storage: createJSONStorage(() => localStorage) — 저장소 지정.
  • skipHydration: true — 핵심. 아래 참고.
💡 typeof window 방어코드가 불필요한 이유 skipHydration: true면 서버 렌더 중엔 localStorage 근처에도 안 간다. 그래서 () => localStorage만 깔끔히 적어도 서버 에러가 안 난다(덕지덕지 방어코드 불필요).

2. 왜 skipHydration이 필요한가 — 불일치의 원리

⚠️ 충돌 시나리오 서버는 개인 localStorage를 못 봄 → 장바구니 0인 정적 HTML 전송. 브라우저는 받자마자 localStorage의 5를 보고 0을 무시하고 5를 그리려 함 → 서버 설계도 ≠ 브라우저 데이터 = Hydration 불일치 에러.

skipHydration: true → 브라우저도 처음엔 서버와 똑같이 0을 그리도록 강제(브레이크). 진짜 데이터 5는 첫 렌더가 평화롭게 끝난 직후 수동 주입.

3. 수동 복구 — Provider의 useEffect + rehydrate()

'use client';
export function CartProvider({ children, initialItems = 0 }) {
  const [store] = useState(() => createCartStore(initialItems));

  useEffect(() => {
    // 바닐라 store엔 persist 타입이 안 보임 → any 대신 정확한 캐스팅
    type PersistedStore = Mutate<StoreApi<CartState>, [["zustand/persist", unknown]]>;
    (store as unknown as PersistedStore).persist.rehydrate(); // 안전한 타이밍에 복구
  }, [store]);

  return <CartContext value={store}>{children}</CartContext>;
}
  • useEffect — 화면이 브라우저에 무사히 그려진 직후에만 실행 → 충돌 없는 시점에 창고 오픈.
  • 타입 캐스팅 — 바닐라 store는 TS가 persist 추가를 모름. any 눈가림 대신 Mutate<StoreApi<T>, [["zustand/persist", unknown]]>로 정확히.
  • use 훅 덕에 커스텀 훅이 undefined 반환 없이 깔끔(예외처리 고통 제거).

4. 남은 깜빡임(0→5) — isMounted 전략

에러는 사라졌지만 새로고침 찰나 0이 보였다 5로 바뀌는 깜빡임이 남는다. 소비 컴포넌트에서 차단:

'use client';
function CartCounter() {
  const [isMounted, setIsMounted] = useState(false);
  const items = useCartStore((s) => s.items);
  useEffect(() => { setIsMounted(true); }, []);

  if (!isMounted) return <div>복구 중...</div>;  // 서버 & 첫 클라 렌더 동일 → 일치
  return <div>{items}</div>;                      // 마운트 후 실제 값
}
원리 서버(항상 false)와 브라우저 첫 렌더(아직 false)가 둘 다 "복구 중..."을 그림 → 100% 일치 → 하이드레이션 성공. 직후 useEffect로 true → 리렌더 → 실제 값. 레이아웃 시프트 방지(스켈레톤으로 공간 확보)도 덤.
📝 과제(강의) 깜빡임 대신 스피너를 더 우아하게 → persist의 onRehydrateStorage 콜백으로 "복구 완료" 플래그를 true로 만들고 그전까지 스피너 분기.

19강. 브라우저와 DB의 실시간 연동

[풀스택 동기화] Server Actions와 낙관적 업데이트(Optimistic UI)

📅 2026-06-21 · 섹션 4 Server ActionsOptimistic UI useTransition롤백
⚡ localStorage만으론 부족(크로스 기기·결제) → Server Actions로 DB 동기화. 핵심은 낙관적 업데이트: 서버 응답을 기다리지 않고 화면부터 즉시 갱신, 백그라운드로 서버 저장, 실패 시 롤백. 롤백은 고정값 덮어쓰기 대신 상대값(-1)으로 해 연속 클릭(race) 방어. 버튼 로딩은 useTransition의 isPending으로.

1. Server Action — 'use server' 보안 터널

'use server';
let mockDB = { cartRecords: [], currentTotal: 0 };

export async function syncCartWithDatabase(items) {
  await new Promise((r) => setTimeout(r, 1000)); // 지연 시뮬레이션
  mockDB.cartRecords.push(items);
  return { success: true };
}
  • 'use server' — 이 파일 함수는 브라우저로 안 내려가고 서버에서만 실행. DB 비밀·민감 로직 안전. API 엔드포인트 없이 일반 함수처럼 호출.
  • 콘솔 위치 — console.log는 브라우저가 아니라 서버(터미널)에 찍힌다.
⚠️ mock DB 함정 메모리 전역 변수라 서버리스(Vercel)에선 요청마다 초기화될 수 있다("배포했더니 장바구니가 자꾸 리셋"). 프로덕션은 실제 DB(prisma.cart.update 등)로 교체 필수.

2. 낙관적 업데이트 (store 액션) — 하이라이트

add: async () => {
  const prev = get().items;          // 1) 원래 값 백업
  set({ items: prev + 1 });          // 2) 화면 '먼저' 즉시 갱신

  const r = await syncCartWithDatabase(prev + 1); // 3) 백그라운드 저장

  if (!r.success) {
    set((s) => ({ items: s.items - 1 }));  // 4) 실패 시 롤백 (상대값!)
  }
}
⭐ 실무 핵심 — 롤백은 "상대값(-1)" 실패 시 set({ items: previousItems })(고정값 덮어쓰기) ❌ → 통신 1초 사이 사용자가 따닥 연속 클릭하면 그 추가분까지 날아감(상태 꼬임). set((s) => ({ items: s.items - 1 }))(현재값에서 -1) ✅ → 연속 클릭(race condition)까지 방어.
왜 store 레벨인가 React 19 useOptimistic 훅도 있지만 컴포넌트 단위. 장바구니처럼 앱 전역 공유 상태의 낙관적 업데이트는 Zustand store 레벨이 더 안정적·확장적. (get()으로 set 밖에서 최신값 참조)

3. Provider — 17·18강과 동일

useState(() => createCartStore()) 지연 초기화 + useEffect에서 persist.rehydrate() + use(CartContext) 전용 훅. (앞 카드 참고)

4. 버튼 — useTransition으로 우아한 로딩

const [isPending, startTransition] = useTransition();

const onClick = () => {
  startTransition(async () => { await add(); }); // 비동기 액션 래핑
};
// 버튼: isPending이면 'SYNCING...' + 투명도, 안 잠금
  • startTransition(async …) — React 19. 비동기 작업을 "급하지 않은 백그라운드"로 표시 → isPending 자동 토글.
  • disabled 생략 — 버튼을 잠그지 않는다. 낙관적 업데이트로 숫자는 즉시 오르고, 버튼은 텍스트·투명도로 로딩만 표시 → 여러 번 자유롭게 클릭 가능.
✅ 흐름 정리 ① 화면 먼저 set(낙관적) → ② await 서버 액션(백그라운드) → ③ 실패 시 상대값 롤백(race 방어). 전역 상태 낙관적 업데이트는 store 레벨, 버튼 로딩은 useTransition. localStorage(개인)와 DB(서버) 두 세계를 Server Action으로 연결.

🔎 정확성 한 줄 패턴은 정확. 단 "-1 롤백"은 각 액션이 독립적 +1일 때만 안전한 단순화 — 실제 서비스는 보통 서버 응답의 실제 수량으로 reconcile(서버를 진실로)하는 게 더 견고. 단일 카운터 예제에선 -1로 충분.

└ @modal/             // 병렬 슬롯 (@ = slot)
   ├ default.tsx      // null 반환 — 평소 빈 슬롯 (필수!)
   └ (.)cart/
      └ page.tsx      // 가로채기 모달 ('use client', router.back())

1. 병렬 라우팅 — @modal 슬롯

  • 보통은 URL 바뀌면 화면 전체 교체. 병렬 라우팅은 한 레이아웃에 독립 화면(슬롯) 여러 개 동시 렌더.
  • 폴더명 앞 @(@modal) → Next가 슬롯으로 인식 → layout의 props로 자동 주입.
export default function RootLayout({ children, modal }) {
  return <CartProvider>{children}{modal}</CartProvider>; // 두 슬롯이 상태 공유
}
⚠️ default.tsx는 바늘과 실 슬롯에 그릴 게 없으면 404로 화면이 깨진다. @modal/default.tsx가 null을 반환해야 평소 슬롯이 투명하게 비고, 가로채기 때만 모달이 뜬다. (병렬 라우팅 쓰면 필수 암기)

2. 가로채기 라우팅 — (.)cart

  • 사용자가 앱 내부 <Link href="/cart"> 클릭 → 전체 이동을 중간에서 낚아챔.
  • (.) = 터미널 ./처럼 "같은 레벨"의 /cart를 가로챈다는 신호.
  • → 실제 /cart/page.tsx 대신 @modal/(.)cart/page.tsx를 모달로 띄움.
가로채기 prefix (.) 같은 레벨 · (..) 한 단계 위 · (..)(..) 두 단계 · (...) 루트 기준.

3. 모달 vs 전체 페이지 — 파일 2개가 필요한 이유

진입 방식보이는 것파일
앱 내부 <Link> 클릭배경 유지 + 모달@modal/(.)cart/page.tsx
주소창 직접 입력 · 새로고침 · 외부 링크전체 화면cart/page.tsx
⚠️ 왜? 직접 접근/새로고침은 앱 내부 이동 궤적을 벗어나 가로채기 컨텍스트가 날아간다. 이때 에러 안 나려면 일반 cart/page.tsx(전체화면)도 반드시 준비. 둘 다 있으면 Next가 상황 보고 알아서 모달/전체 선택.

4. 닫기 — router.back()의 특별함

가로채기 모달에서 router.back()은 일반 뒤로가기와 다르다:

  • 주소창 URL만 모달 뜨기 전으로 되돌림
  • 배경 페이지는 재렌더 없이 그대로 유지, 모달 슬롯만 비움 → 모달만 깔끔히 닫힘

📏 레이아웃 시프트 방지

21강. 다크모드 테마 관리

[테마 관리] Zustand Persist + Tailwind CSS v4로 구현하는 다크모드

📅 2026-06-21 · 섹션 4 (마지막) FOUCmatchMedia Tailwind v4store.subscribe
🌗 테마(light/dark/system)를 persist로 저장하고, Provider의 useEffect에서 rehydrate + <html> classList 조작 + store.subscribe로 DOM 동기화. system은 matchMedia로 OS 감지. Tailwind v4는 @custom-variant dark로 .dark 클래스 수동 제어. <html suppressHydrationWarning>로 클래스 주입 mismatch 경고 억제.

0. 문제 — FOUC (화면 깜빡임)

⚠️ 왜 생기나 서버는 유저 테마를 모름 → 기본 light HTML 전송 → 브라우저가 localStorage 확인하니 dark → 뒤늦게 검정으로 전환. 이 0.1초 하얀 번쩍임이 FOUC(Flash of Unstyled Content).

1. 스토어 — 18강 persist와 동일 구조

type Theme = 'light' | 'dark' | 'system';

createStore<ThemeState>()(persist(
  (set) => ({ theme: 'system', setTheme: (t) => set({ theme: t }) }),
  { name: 'theme-storage', storage: createJSONStorage(() => localStorage),
    skipHydration: true }, // 자동 주입 OFF (mismatch 방지)
));

2. Provider — DOM 동기화가 핵심

useEffect(() => {
  store.persist.rehydrate();                  // localStorage 복구

  const updateDom = (theme) => {
    const root = document.documentElement;     // <html>
    root.classList.remove('light', 'dark');
    const t = theme === 'system'
      ? (matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light')
      : theme;
    root.classList.add(t);                     // <html class="dark">
  };

  updateDom(store.getState().theme);
  return store.subscribe((s) => updateDom(s.theme)); // 변경 감지 → DOM 갱신
}, []);
  • 테마는 React state가 아니라 <html> classList로 반영 → Tailwind dark: 발동.
  • store.subscribe — Zustand 바닐라 store의 React 밖 구독. 상태 변하면 콜백으로 DOM 직접 갱신. (앞 인사이트의 "React 밖 접근"의 실사용 예)
  • matchMedia('(prefers-color-scheme: dark)') — OS 다크모드 감지(system 모드).

3. Tailwind v4 — @custom-variant dark (필수)

/* globals.css */
@import "tailwindcss";
@custom-variant dark (&:where(.dark, .dark *));
왜 필요 Tailwind v4는 기본적으로 OS 설정만 따른다 → <html>에 .dark를 붙여도 무시됨. @custom-variant dark로 ".dark 클래스가 있으면 OS와 무관하게 강제 다크"라고 선언해야 수동 제어 가능.

4. layout — 배치 2가지 실무 팁

  • ThemeProvider를 최외곽에 — 테마는 UI 근간이라 CartProvider 등 다른 Provider보다 바깥.
  • <html suppressHydrationWarning> — 클라에서 테마 클래스가 동적 주입돼 서버 HTML과 달라지는 mismatch 경고를 억제(html 태그 한정, 정당한 용법 — 18강 팩트체크 참고).
✅ 정리 persist(skipHydration) → useEffect에서 rehydrate + html classList + store.subscribe → 새로고침에도 유지·버튼 즉시 전환. system은 matchMedia, Tailwind v4는 @custom-variant dark, html엔 suppressHydrationWarning.
🔎 정확성 — "깜빡임 없는"은 과장 이 패턴은 하이드레이션 에러는 없애지만 FOUC(깜빡임) 자체는 안 없앤다. useEffect는 첫 페인트 "이후" 실행 → dark 유저는 여전히 light→dark 번쩍임을 본다. 진짜 flash-free는 <head>에 blocking inline script로 페인트 전에 클래스를 박는 방식(next-themes가 이걸 함). 추가로 matchMedia에 change 리스너가 없어 OS 테마 실시간 변경은 반영 안 됨(읽기 1회).

15강 팩트체크 — 과장·누락 보정

📅 2026-06-21 팩트체크useShallowRSC
Q. 15강 주장 중 기술적으로 틀리거나 오해를 부를 부분은 없나?
🔎 큰 그림(확성기 효과·관료주의·클라 상태=Zustand)은 옳다. 단 비유 위주라 생긴 과장·누락이 있다. 특히 ① "Provider가 RSC를 해친다"는 틀림, ② selector 객체 반환 시 useShallow 필요(누락 시 치명), ③ "전역 store=무조건 누수"는 과장.

🔴 명백히 틀림

"Context Provider로 감싸면 RSC 이점이 사라진다"

  • <Provider>{children}</Provider>처럼 children을 prop으로 끼우면 Provider가 'use client'여도 children은 서버 컴포넌트로 렌더된다.
  • RSC가 깨지는 건 Provider가 아니라 실제 useContext로 소비하는 컴포넌트를 클라이언트로 만들 때.
  • "Provider는 루트에 둬야 한다"도 거짓 — leaf로 내릴 수 있다.

🟡 오해 소지 — 주석 필요

강의 주장보정
"selector로 자동 최적화"올바른 selector를 줘야 함. 객체/배열 반환 시 매번 새 참조 → useShallow 필수. selector 없이 전체 구독하면 Context처럼 다 리렌더
"값 하나만 바뀌어도 전체 리렌더"정확히는 value 참조가 바뀌면. Context도 잘게 쪼개면 완화 가능
"set은 자동 병합"얕은(top-level) 병합만. 중첩 객체는 수동, set(_, true)면 통째 교체
"전역 store = 누수"과장. 서버 렌더 경로에서 store에 write/요청별 데이터 주입할 때만 누수·hydration 문제. 순수 클라 UI 상태(모달 토글)는 모듈 store도 안전
"RTK도 관료주의 그대로"store/Provider 구조는 남지만 createSlice로 작성 경험은 크게 개선됨. 과소평가
"구독 컴포넌트만 use client" = Zustand 장점맞지만 Zustand 전용 아님 — Context도 동일
⚠️ 가장 실수하기 쉬운 함정 — useShallow 
// ❌ 매 렌더 새 객체 → Object.is 항상 false → 항상 리렌더
const { a, b } = useStore((s) => ({ a: s.a, b: s.b }));

// ✅ 얕은 비교
import { useShallow } from 'zustand/react/shallow';
const { a, b } = useStore(useShallow((s) => ({ a: s.a, b: s.b })));
단일 원시값(s.count)만 뽑으면 함정 없음. 문제는 "여러 값을 객체/배열로 묶을 때".

🟢 정확한 것

  • Provider에 변하는 값이 아니라 고정 store 인스턴스만 담아 확성기 회피 ✅
  • createStore(vanilla, 순수) vs create(React 훅 결합) 구분 ✅

16강 팩트체크 — 비유가 단순화한 지점

📅 2026-06-21 팩트체크SSRProvider
Q. 16강 주장 중 틀리거나 오해를 부를 부분은? (15강보다 정확한 편)
🔎 명백히 틀린 건 없다. 권장 패턴(요청별 store + 클라 Provider + useRef + selector)은 Next.js/Zustand 공식 가이드와 일치. 다만 비유가 ① 리렌더 메커니즘, ② 누수 조건, ③ "서버가 store를 생성"이라는 표현을 단순화했다.

🟢 정확한 것 (핵심 패턴은 옳음)

  • set 안 쓰고 직접 mutate → 리렌더 안 됨 ✅ (메커니즘 보강은 아래)
  • 요청마다 store 새로 생성 → Provider 전달 ✅ (공식 권장)
  • Provider엔 store 인스턴스(명함)만 → 폭포수 없음 ✅
  • useRef + if(!storeRef.current)로 1회 생성 ✅

🟡 비유가 단순화 — 주석 필요

강의 주장정확한 보정
"set이 리렌더 신호를 보낸다" 정확히는 set이 내부 상태 교체 후 구독 listener를 notify(useSyncExternalStore). 또 set을 써도 객체를 in-place mutate하면 같은 참조 → Object.is가 "안 변함" 판정 → 리렌더 누락. 핵심은 "set" 자체가 아니라 불변(새 참조) 업데이트 + notify
"함수형 업데이트는 useState와 같은 원리" 결과는 같지만 원리는 다름. useState는 stale closure 회피용이 핵심. Zustand는 get()으로 항상 최신을 읽어 set({count: get().count+1})도 stale 없음 → 함수형이 "필수"는 아님
"자동 병합" 1레벨 얕은 병합만. 중첩(user:{name,age})에서 set({user:{name}})하면 age 소멸 → ...state.user 전개 필요. set(_, true)면 전체 교체
"useState/context는 모든 자식 폭포수" props drilling은 대체로 맞지만, Context는 consumer만 리렌더(memo·value 메모이즈·context 분리로 완화). "무조건 모든 자식"은 과장
"전역 store = 누수" 조건부. ① 모듈 스코프 싱글톤 + ② 서버 렌더 경로에서 그 store에 read/write가 겹칠 때 누수. 클라에서만 mutate하면 누수 아님(이땐 하이드레이션 문제)
"서버가 사용자별 store를 생성" 생성 주체는 클라이언트 Provider. 실제 흐름: 서버 컴포넌트는 데이터만 fetch → initialState prop으로 전달 → 클라 Provider가 useRef로 store 생성. store 인스턴스(함수/클래스)는 서버→클라 직렬화 불가라 RSC가 직접 만들어 내려보낼 수 없음
⚠️ 강의가 빠뜨리기 쉬운 필수 제약 이 Provider는 useRef/useState/createContext를 쓰므로 반드시 'use client' 컴포넌트. 순수 서버 컴포넌트(RSC)에선 이 훅들을 못 쓴다. → "서버에서 만든다"가 아니라 "서버는 데이터, 생성·격리는 클라 Provider의 렌더 스코프"가 정확한 그림.
왜 useRef가 요청별 격리를 보장하나 useRef는 모듈 전역이 아니라 컴포넌트 인스턴스(렌더 스코프) 지역. 요청마다 독립 React 트리에서 Provider가 새로 렌더 → useRef도 새로 초기화 → 격리. 핵심은 "모듈 스코프가 아니라 렌더 스코프".
✅ 한 줄 결론 16강의 권장 아키텍처는 정확하다. 단 비유에서 "set=신호"(→ 불변+notify), "누수=무조건"(→ 조건부), "서버가 store 생성"(→ 클라 Provider 생성) 세 가지만 정정하면 완벽. + 하이드레이션: 서버/클라가 같은 initCount면 mismatch 없음. persist(localStorage)가 끼면 별도 처리 필요(다음 강).

📅 2026-06-21 팩트체크PPRskipHydration
Q. 18강 주장 중 틀리거나 인과를 잘못 짚은 부분은? (해결 패턴 자체는 맞음)
🔎 핵심 패턴(skipHydration + 수동 rehydrate + isMounted)은 실무적으로 정확. 하지만 ① PPR 주장은 3겹으로 틀렸고, ② "skipHydration이 서버 에러를 막는다", ③ "use 훅이 undefined를 없앤다"는 인과를 잘못 짚었다.

🔴 명백히 틀림 — PPR 주장

"Next 15부터 PPR 기본화 → 응답이 빨라 불일치가 더 빈번 → skipHydration 필수"

  • 기본화 아님 — PPR은 Next 15에서 experimental opt-in(experimental.ppr:'incremental' + 라우트별 experimental_ppr). 디폴트가 아니다.
  • 인과 오류 — 이 불일치는 PPR과 무관. 클라 전용 localStorage + 서버 프리렌더(SSR/SSG/ISR 무엇이든)면 항상 나는 결정론적 현상(서버는 localStorage를 못 봄).
  • "빨라서 더 난다"는 틀림 — hydration mismatch는 렌더된 마크업 차이로 판정되지 타이밍·속도와 무관.
🔬 정밀 보강 — "Next 16에서 hydration 에러가 늘었다"는 체감의 진짜 정체 강의의 PPR 인과는 틀렸지만, "요즘 더 자주 터진다"는 체감 자체엔 실제 원인이 있다. 단 PPR이 본질이 아니라 Next 16이 묶어온 변화들의 복합 효과:
  • 진짜 주범 = React 19 에러 정책 (PPR 아님) — React 18은 mismatch를 warning으로 흘렸지만 React 19는 hard error로 던진다. Next 16이 React 19를 기본 탑재 → "버그가 늘었다"가 아니라 묻혀 있던 게 빨갛게 드러났다."
  • PPR의 구조적 기여 (이 부분은 팩트) — PPR은 정적 셸을 먼저, 동적 부분을 나중에 스트리밍. 사용자별 데이터(이름·장바구니 수·타임스탬프)를 Suspense로 안 감싸면 서버는 빈 셸, 클라는 데이터 기대 → mismatch. 업그레이드 후 가장 흔한 보고.
  • 회피 가능 — 처방은 명확: 동적 데이터를 fallback 일치하는 Suspense 경계로 감싸기. 제대로 하면 PPR이 오히려 정적/동적을 깔끔히 분리. (Next 16 에러 오버레이는 + Client / - Server diff로 갈린 쪽을 바로 표시)

Q. 21강의 다크모드 패턴은 정말 "깜빡임 없는" 완벽한 구현인가?
🔎 정적 구성(Tailwind v4 @custom-variant, persist 문법)은 정확. 하지만 셀링포인트인 "깜빡임 없음"이 거짓 — 이 패턴은 hydration 에러만 없애고 FOUC(flash)는 그대로 남긴다. 또 matchMedia에 change 리스너가 없어 OS 테마 실시간 반영도 안 됨.

🔴 명백히 틀림 — "깜빡임 없는 완벽한 다크모드"

hydration 에러 해결 ≠ flash 해결 (둘을 혼동)

렌더 타임라인:

  1. SSR: <html>에 테마 클래스 없음(서버는 유저 테마 모름)
  2. 브라우저 첫 페인트: 클래스 없이 light로 실제 그려짐
  3. hydration 후 useEffect 실행 → 그제야 .dark 추가
  4. → dark 유저는 light→dark 번쩍임을 그대로 봄

useEffect는 페인트 이후(commit 후) 실행 → 페인트 전에 클래스를 못 박는다. "깜빡임 없음"은 성립 불가.

✅ 진짜 flash-free 조건 <head>에 blocking inline script(렌더 차단)를 넣어 페인트 전에 동기적으로 localStorage를 읽고 documentElement.classList를 세팅해야 한다. next-themes가 ThemeScript로 하는 게 바로 이것. 강의 코드엔 이 inline script가 없으므로 flash-free 아님.

🔴 틀림 — "matchMedia로 OS 실시간 판별"

change 리스너가 없다

  • matchMedia(...).matches를 1회 읽기만 함. .addEventListener('change', …) 없음.
  • → system 모드에서 런타임에 OS 테마를 바꿔도 화면 자동 갱신 안 됨(새로고침 전까지 옛 테마).
  • store.subscribe는 Zustand 상태 변화만 감지 — OS 미디어쿼리 변화는 못 잡음. "실시간"은 과장.

🟡 오해 소지 — 주석 필요

강의 주장보정
"html에 suppressHydrationWarning으로 클래스 mismatch 억제" 용법 자체는 정당(next-themes도 씀). 단 이 코드는 클래스 주입이 useEffect(hydration 이후)라 애초에 mismatch가 없음 → 실질 잉여. 정작 필요해지는 건 빠져있는 inline-script 방식일 때
"skipHydration으로 렌더 불일치 방지" 일반론은 맞지만, 이 코드는 theme를 JSX에 렌더 안 하고 classList 명령형 조작만 함 → 막아줄 렌더 mismatch가 사실상 없음(과한 정당화)
"ThemeProvider는 근간이라 최외곽에 둬야" 합리적 관례지만 강제 아님. documentElement 명령형 조작이라 Provider 중첩 순서와 기술적으로 무관(안쪽에 둬도 <html>에 똑같이 붙음)
(17강) useState 지연초기화 권장 → 21강 useRef 회귀 둘 다 React 공식 인정 패턴이라 안전성 차이는 실질 없음. 단 같은 시리즈에서 권장안 번복은 교육적 일관성 결여

🟢 정확한 것

  • FOUC 발생 원리 설명 ✅ (단 제시 코드가 그걸 못 고치는 게 모순)
  • Tailwind v4 기본은 OS만 따름 → @custom-variant dark (&:where(.dark, .dark *))로 클래스 수동 제어 ✅ (공식 구문, :where는 명시도 0 유지)
  • createStore<T>()(persist(...)) 커리드 시그니처 ✅
  • store.subscribe로 React 밖에서 DOM 갱신 ✅ ("React 밖 접근"의 실사용)
✅ 총평 정적 구성은 정확하나 핵심 셀링포인트 "깜빡임 없음"이 거짓. 실제론 hydration 에러만 제거, FOUC는 남음. 빠진 두 장치 = ① head의 blocking inline script(진짜 flash-free) ② matchMedia change 리스너(OS 실시간). 둘 다 next-themes는 구현 → 실무는 그냥 next-themes 권장.

지연 초기화 (Lazy Initialization)

📅 2026-06-21 useStateReact 16.8+store 생성
Q. 17·18강의 const [store] = useState(() => createCartStore(...)) — 왜 값이 아니라 함수를 넣나? 언제·왜 생긴 문법이고 원리·주의점은?
🥚 useState에 값 대신 함수를 주면 React가 그 함수를 최초 마운트 1회만 호출해 초기값을 얻고, 이후 렌더에선 완전히 무시한다. 비싼 계산·인스턴스 생성을 매 렌더 반복하지 않기 위한 것. 핵심은 "값을 지금 계산" vs "계산 방법을 넘겨 React가 한 번만".

1. 무엇이 문제인가 — 값 형태의 낭비

❌ 즉시 평가 (매 렌더 실행)

const [s] = useState(createStore());
// createStore()가 '매 렌더' 실행됨.
// 결과는 첫 렌더만 쓰고 나머진 버림 → 낭비

✅ 지연 초기화 (1회 실행)

const [s] = useState(() => createStore());
// 화살표 함수 '몸통'은 최초 1회만 실행
// 이후 렌더에선 React가 인자를 무시
왜 값 형태가 낭비인가 useState(x)에서 React는 리렌더 시 인자 x를 무시(이미 state를 들고 있으니)하지만, JS는 함수에 넘기기 전 x 표현식을 항상 평가한다. createStore()가 거기 있으면 매 렌더 호출된다 — 결과는 안 쓰는데도.

2. 언제·왜 도입됐나

  • 시점 — Hooks와 함께 React 16.8 (2019.2). "lazy initial state"는 처음부터 useState 명세에 포함.
  • 이유 1 (성능) — 무거운 초기 계산(JSON.parse(localStorage…), 큰 배열 생성)을 매 렌더 반복하지 않으려고.
  • 이유 2 (정확성/정체성) — new QueryClient()·createStore()처럼 인스턴스를 딱 하나만 만들어야 하는 경우. 매 렌더 새로 만들면(설령 버려져도) 비용·부작용 발생.

3. 원리 (React가 하는 일)

useState는 인자가 함수면 "초기화 함수(initializer)"로 취급한다.

  • 최초 마운트 — initializer를 인자 없이 호출해 반환값을 초기 state로 저장.
  • 리렌더 — 이미 저장된 state가 있으므로 initializer를 호출하지 않고 인자 자체를 무시.
참고 useReducer의 지연 초기화는 형태가 다르다 → useReducer(reducer, initialArg, init)에서 init(initialArg)로 호출(인자 받음). useState initializer는 인자를 받지 않는다.

4. 어떻게 쓰나

// 무거운 초기 계산
const [tree] = useState(() => buildHugeTree(props.seed));
// localStorage 파싱
const [draft] = useState(() => JSON.parse(localStorage.getItem('draft') ?? '{}'));
// 인스턴스 1회 생성 (Zustand/React Query)
const [store]  = useState(() => createCartStore(initialItems));
const [qc]     = useState(() => new QueryClient());

5. ⚠️ 주의점

흔한 함정

  • 함수를 "호출"하면 무의미 — useState(createStore())(❌) vs useState(() => createStore())(✅). 괄호 하나 차이로 매 렌더 실행.
  • initializer는 1회뿐 → props 변경에 반응 안 함 — useState(() => createStore(initialItems))는 첫 initialItems만 박제. 이후 prop이 바뀌어도 store는 안 바뀐다(파생상태 함정). SSR 패턴에선 보통 의도된 동작.
  • 순수해야 함 — initializer 안에서 구독·이벤트 등록 같은 부작용 금지. React 18+ StrictMode(dev)는 initializer를 2번 호출해 불순물을 노출시킨다(인스턴스가 dev에서 2개 생길 수 있음 → 순수 생성이면 OK).
  • "1회" = per mount, 영원히 1회 아님 — 리렌더엔 안 돌지만 언마운트 후 재마운트엔 다시 실행(새 인스턴스 = 새 state). key 변경·조건부 렌더 제거/복귀·라우트 이동·StrictMode가 재마운트를 유발. 마운트를 넘는 보존이 필요하면 그건 지연초기화가 아니라 persist / 상태 끌어올리기 / 외부 store의 몫.
📌 재마운트 시 동작 (Zustand 맥락) CartProvider가 언마운트→재마운트되면 새 store 인스턴스가 생겨 인메모리 상태는 사라진다. 단 persist면 rehydrate()가 localStorage에서 값을 다시 끌어와 사용자 눈엔 보존(인스턴스는 새것, 영속 데이터는 유지).

⚠️ useMemo로 "1회 생성" 하지 말 것 useMemo(() => createStore(), [])는 정체성 보장이 아니다 — React는 메모리 압박 시 캐시를 버리고 재계산할 수 있다(공식 문서). "딱 하나만"이 필요하면 useState 지연 초기화나 useRef 가드를 써라.

6. useState 지연초기화 vs useRef 가드 (17강 맥락)

useRef + 가드 (과거 패턴)

const ref = useRef();
if (!ref.current)
  ref.current = createStore();

useRef(createStore())는 매 렌더 생성되므로 반드시 if 가드 필요.

useState 지연초기화 (권장)

const [store] =
  useState(() => createStore());

가드 불필요. React 19 동시성에서 렌더가 중단·재시작돼도 React가 state로 안전 관리 → 17강이 useRef에서 이걸로 전환한 이유.

✅ 한 줄 값을 넣으면 매 렌더 계산, 함수를 넣으면 최초 1회. 비싼 계산·단 하나의 인스턴스엔 useState(() => ...). 단 initializer는 순수·1회뿐(props 변경 무반응, dev 2회 호출), useMemo로 정체성 보장 금지. 그리고 "렌더 사이 안정성"이지 "마운트를 넘는 영속"이 아니다 — 재마운트엔 다시 돈다.

⚡ DB가 진실이면 그건 서버 상태 → useMutation이 정석(낙관적+롤백+재동기화 내장). 강의의 Zustand 손코딩은 화면만 맞추고 서버 정합은 빈 채로 둔다. 연속 클릭 시 화면은 정상 누적(get 최신값+동기 set)되지만, 서버는 N개 동시요청 + 완료 순서 race로 틀어질 수 있다. -1 롤백은 그래서 짠 것.

1. 연속 클릭 추적 — 화면은 왜 정상인가

add가 매번 get().items(최신값)를 읽고 set을 await 전에 동기 실행하기 때문:

클릭get().itemsset서버요청
1 (t=0)0items=1sync(1)
2 (t=50ms)1 (1번 set 반영)items=2sync(2)
3 (t=100ms)2items=3sync(3)
그래서 화면은 0→1→2→3 정확히 누적 — 핸들러를 안 막아도 lost update 없음. (만약 prev를 클로저 캡처하거나 set 전에 await가 끼면 누락 났을 것). isPending은 하나라도 진행 중이면 true라 버튼은 내내 'SYNCING' + 투명도, 단 disabled 아니라 클릭은 계속 먹힘.

2. 숨은 비용 2가지

① N개 동시 요청 (dedup/취소 없음)

3번 누르면 sync(1/2/3)이 겹쳐서 발송. startTransition은 중복제거도 이전요청 취소도 안 함.

② 서버 측 race (진짜 함정)

3 요청이 각각 currentTotal=items. 완료 순서 미보장 → sync(3)이 먼저 끝나면 최종 DB가 3이 아니라 1. 화면 3 ≠ DB 1 불일치.

3. 그래서 롤백이 "-1 상대값"인 이유

클릭 2만 실패 시(이미 items=3): set(s => s.items - 1) → 2.

  • 카운터는 어느 +1을 빼든 동등(fungible) → 결과 2 = "성공 2건"과 일치 ✅
  • set({items: previousItems})(=1)로 덮으면 클릭 3 증가분까지 소실 ❌
한계 -1 롤백은 클라 일관성만 지킨다. 위 서버 측 race(2번)는 여전히 미해결 — 화면과 DB가 갈릴 수 있음.

4. 왜 그냥 TanStack Query인가 — 진실의 원천으로 판단

DB에 저장돼 그게 진실의 원천이면(로그인·결제·크로스기기) = 서버 상태 → useMutation이 이걸 위한 도구. 강의는 자기 15강 원칙("서버상태=TanStack")을 어겼다.

useMutation({
  mutationFn: syncCartWithDatabase,
  onMutate: async (next) => {
    await qc.cancelQueries({ queryKey: ['cart'] }); // 중복 정리
    const prev = qc.getQueryData(['cart']);          // 전체 스냅샷
    qc.setQueryData(['cart'], next);                  // 낙관적
    return { prev };
  },
  onError: (_e, _v, ctx) => qc.setQueryData(['cart'], ctx.prev), // 롤백
  onSettled: () => qc.invalidateQueries({ queryKey: ['cart'] }), // 서버 진실로 재동기화
});
왜 더 견고한가 ① onMutate가 전체 이전 스냅샷 복원(부분 실패 안전) · ② onSettled의 invalidate→refetch로 위 서버 race를 서버 진실에 맞춰 reconcile(손계산 -1 불필요) · ③ 캐싱·dedup은 덤. 강의의 "-1"이 못 한 걸 구조적으로 해결.

5. 결정 규칙 & 우리 프로젝트

✅ "진실의 원천이 누구냐"로 갈린다
· 서버가 진실(로그인·결제·크로스기기) → TanStack Query useMutation(낙관적 내장)
· 클라가 진실(local-first·즉각·오프라인, DB는 fire-and-forget) → Zustand/Jotai + persist + 백그라운드 sync(강의 방식)
· 현실 카트는 흔히 하이브리드(게스트=클라 / 로그인=서버)
📦 우리 프로젝트 이미 TanStack Query ~1,600 사용처로 서버 상태 관리 → DB 동기 카트면 당연히 useMutation, jotai/zustand로 끌고 갈 이유 없음. 강의의 Zustand-옵티미스틱은 패턴 학습용(낙관적 원리 자체는 라이브러리 무관 — jotai write-atom으로도 동일).

Jotai vs Zustand — '값이 어디 사는가'로 이해하기

📅 2026-06-21 JotaiZustand멘탈모델SSR
Q. 둘 다 pmndrs 라이브러리인데 뭐가 본질적으로 다르고, 왜 Next.js SSR에서 다르게 동작하나? (표만 봐선 이해가 안 됨)
🔑 딱 하나만 기억하면 나머지가 다 풀린다 → "값(state)이 어디에 저장되는가". Zustand는 store 객체가 값을 직접 들고 있다(정의=그릇=값). Jotai는 atom은 '이름표(키)'일 뿐 값이 없고, 값은 별도의 store에 따로 산다(정의 ↔ 값 분리). 이 한 줄에서 리렌더·SSR·보일러플레이트 차이가 전부 갈린다.

0. 단 하나의 핵심 — 값의 거처

🐻 Zustand — 그릇이 값을 들고 있음

create()가 만든 store는 그 자체가 값을 담은 그릇이다. "정의 + 값"이 한 덩어리.

const useStore = create(() => ({ count: 0 }));
// useStore 안에 0이 '실제로' 들어있음
useStore.getState().count; // 0

⚛️ Jotai — 이름표 + 창고 분리

atom()은 값이 없는 '이름표(키)'. 실제 값은 store(창고)에 저장되고, atom은 그 칸을 가리키는 주소일 뿐.

const countAtom = atom(0);
// countAtom엔 0이 없다. '초기값 0짜리 칸'의 이름표일 뿐
store.get(countAtom); // store(창고)가 값을 보관
🗄️
Zustand = 내용물이 든 도시락 통(통=값). 통을 복제하면 내용물도 복제됨. Jotai = 사물함 번호표(atom) + 사물함 건물(store). 번호표는 어디에 붙여도 안전하고(값이 없으니), 실제 짐은 "어느 건물의" 그 번호 칸에 있느냐로 정해진다. → 같은 번호표도 건물(store)이 다르면 다른 짐.

1. 같은 기능을 양쪽으로 — 코드로 체감

🐻 Zustand

// 1) 그릇(상태+액션)을 통째로 정의
const useStore = create<S>((set) => ({
  count: 0,
  name: 'kim',
  inc: () => set((s) => ({ count: s.count+1 })),
}));

// 2) selector로 '슬라이스'를 잘라 구독
const count = useStore((s) => s.count);
const inc   = useStore((s) => s.inc);

큰 그릇 하나를 만들고 잘라서 쓴다 (top-down).

⚛️ Jotai

// 1) 조각(atom)을 따로따로 선언
const countAtom = atom(0);
const nameAtom  = atom('kim');

// 2) 필요한 atom만 집어 구독
const [count, setCount] = useAtom(countAtom);
const inc = () => setCount((c) => c+1);

작은 조각을 모아서 쓴다 (bottom-up). "전역 useState" 느낌.

2. 리렌더 — 둘 다 "필요한 것만"이지만 방식이 다르다

ZustandJotai
구독 단위selector가 잘라낸 결과atom 그 자체
비교 방식이전·이후 결과를 Object.is구독한 atom이 바뀌었나(그래프)
흔한 함정객체 반환 시 매번 새 참조 → useShallow 필요atom 단위라 참조 함정이 적음
파생 상태selector 안에서 계산(구독 단위는 원본)derived atom(의존 atom 변할 때만 재계산·캐시)

🐻 Zustand 함정

// 매 렌더 새 객체 → 항상 리렌더
const {a,b} = useStore((s)=>({a:s.a,b:s.b}));
// ✅ useShallow로 해결
useStore(useShallow((s)=>({a:s.a,b:s.b})));

⚛️ Jotai 파생

const countAtom   = atom(0);
// count 바뀔 때만 재계산, 결과 캐시됨
const doubledAtom = atom((get)=>get(countAtom)*2);

직관: 상태가 잘게 독립적이면 Jotai가 자연스럽게 세밀(조각 단위가 기본값). 하나의 큰 트리를 여러 각도로 슬라이싱하면 Zustand selector가 유연. 우열이 아니라 "기본값에서 실수 가능성 + 보일러플레이트"의 차이.

3. Next.js SSR에서 왜 다르게 동작하나 (가장 중요)

⚠️ 공통 전제 Next 서버는 한 프로세스로 모든 요청을 처리한다. 모듈 스코프(파일 최상단)에 만든 값을 담은 그릇은 요청끼리 공유 → 서버 렌더 중 값을 쓰면 사용자 간 누수. 그래서 "값을 담은 것"을 모듈 전역에 두면 안 된다.

🐻 Zustand — store = 값 그릇 → 전역 금지

// ❌ 모듈 전역 = 값이 서버에 공유됨
export const useStore = create(...);

// ✅ 요청마다 새로 찍는 '공장' + 내 Context
const makeStore = (init) =>
  createStore(() => ({ count: init }));
// Provider에서 useRef로 1회 생성해 격리

store가 곧 값이라 전역에 못 둔다 → 공장 함수 + 커스텀 Context + useStore(store, sel) 직접 배선. 보일러플레이트 ↑

⚛️ Jotai — atom = 값 없는 키 → 전역 OK

// ✅ atom은 값이 없으니 전역에 둬도 안전
export const countAtom = atom(0);

// 값 격리는 Provider(요청별 store)가 담당
<Provider>{children}</Provider>
// 서버 초기값 주입
useHydrateAtoms([[countAtom, serverValue]]);

atom 정의는 전역 그대로 두고, 값만 <Provider>가 만든 store에 격리. 공장 함수 불필요. 보일러플레이트 ↓

정확히 짚기 (오해 방지) "Jotai라서 Provider가 필요 없다"가 아니다. SSR 격리·초기값 주입을 하려면 Jotai도 <Provider>를 쓴다(안 쓰면 모듈 전역 default store를 쓰는데, 그건 서버에서 공유됨 → 같은 누수 위험). 진짜 차이는 "무엇을 전역에 둘 수 있나"다:
· Zustand: 값 그릇(store)을 전역에 두면 위험 → 공장+Context 직접 구성 · Jotai: 이름표(atom)는 전역 안전, 값 격리는 Provider가 기본 제공 → 더 가벼움
한 문장 둘 다 Provider로 요청별 값을 격리한다(둘 다 Provider엔 변하는 값이 아니라 store 참조만 담아 폭포수 없음). 차이는 Jotai는 atom(정의)이 값과 분리돼 전역에 둬도 안전해 배선이 적고, Zustand는 store가 값을 들고 있어 요청별 공장+Context를 직접 짜야 한다는 것.

📌 "React 밖 접근"이란? 컴포넌트·훅이 아닌 평범한 JS(웹소켓 socket.on 콜백, 전역 addEventListener, setInterval, store 액션 내부, 테스트). 여기선 훅(useStore(selector))을 못 쓴다. Zustand store는 훅이자 평범한 store 객체라 밖에서 getState()/setState()로 직접 접근된다.
⚠️ 먼저 — 이건 store에 둘 게 아니다 토큰·테마·언어·공유할 필터처럼 영속·공유·직렬화 상태는 store가 아니라 쿠키 / localStorage / 쿼리스트링이 정답(store에만 두면 새로고침에 날아감 = 안티패턴). 그래서 "axios 인터셉터에서 토큰 읽기"는 보통 쿠키(httpOnly면 자동 전송)나 localStorage로 해결 — store가 필요 없다.

결정 규칙 — 어디에 둘까

🍪 브라우저 상태 (쿠키 / localStorage / URL)

  • 영속(새로고침·재방문에도 유지): 토큰·테마·언어
  • 공유·북마크·뒤로가기: 검색어·필터·탭 → 쿼리스트링
  • 직렬화 가능한 작은 문자열

🐻 인메모리 store (Zustand)

  • 휘발성(영속하면 안 됨): 모달 열림·진행 중 폼
  • 반응형 + 크거나 잦은 변경: 실시간 스트림·큰 파생 데이터
  • URL/스토리지에 담기 부적절한 런타임 객체

그래서 'React 밖 접근'이 진짜 필요한 경우 (브라우저 상태로 못 가는 것)

공통 조건: (a) 영속·공유 불필요 + (b) 인메모리·반응형 + (c) 트리거가 React 밖. 셋이 겹칠 때만.

  • 실시간 스트림 — 웹소켓/SSE로 쏟아지는 채팅·알림·시세를 메모리에 쌓기. localStorage 부적합(용량·성능·반응성). 소켓 콜백은 React 밖 ✅
  • 휘발성 UI 상태를 밖에서 읽기 — 전역 단축키 리스너가 getState().isModalOpen 확인. 모달 열림은 영속하면 안 되는 상태 → 쿠키/스토리지가 틀림 ✅
  • 액션 간 연쇄 — store 액션 안에서 get()으로 다른 슬라이스를 읽어 계산. 브라우저 상태와 무관, 대체재 없음 ✅
  • 테스트 — 컴포넌트 렌더 없이 setState()로 상태 세팅 후 로직만 검증 ✅
// socket.ts  ← React 밖 콜백 + 휘발성·반응형 스트림 = store가 정답
socket.on('message', (msg) => {
  useChatStore.getState().addMessage(msg);   // 밖에서 직접 push
});

// shortcut.ts  ← 전역 리스너가 '지금 휘발성 UI 상태'를 읽음
window.addEventListener('keydown', (e) => {
  if (e.key === 'Escape' && useUIStore.getState().isModalOpen) {
    useUIStore.getState().closeModal();
  }
});
🔗 둘은 양자택일이 아니다 localStorage는 같은 탭에서 변경이 반응형이 아니라서, 영속값을 store에 미러링해 UI 반응성을 얻는다. Zustand persist가 이 동기화(store ↔ localStorage)를 자동화. → 예: 테마 = 저장은 localStorage · 화면 반응은 store · 연결은 persist.
⚠️ getState() 주의 구독이 아니라 그 순간 스냅샷. "바뀌면 다시 실행" 안 됨. → 화면에 그릴 값은 컴포넌트 안 useStore(selector)(구독), 밖에서 명령형으로 '지금 값만' 필요할 때 getState(). 컴포넌트 안에서 getState() 쓰면 리렌더 안 되니 금물.

6. 언제 무엇을

🐻 Zustand가 유리

  • 인증·테마·장바구니 같은 명확한 도메인 store 하나
  • "store + 액션" Redux형 멘탈모델 선호
  • React 밖(이벤트 핸들러·유틸)에서 상태 접근
  • 미들웨어를 중앙에서 명시적으로 합성

⚛️ Jotai가 유리

  • 상태가 잘게 쪼개진 독립 단위, 로컬→전역 승격 잦음
  • 파생 상태가 많고 의존 그래프 자동 재계산 필요
  • App Router SSR을 최소 보일러플레이트로
  • 비동기 + Suspense를 상태와 매끄럽게 통합
⚖️ 솔직한 무승부 — "능력 차이"는 거의 없다 위 "유리" 항목은 capability가 아니라 결(shape)·취향·생태계 적합성이다. 깐깐히 따지면:
  • 응집도(상태+액션 한곳) — Jotai도 atom + write-atom(atom(null,(get,set)=>...))을 한 모듈/훅에 모으면 동일. 오히려 파생 상태는 Jotai가 더 응집적(derived atom 자동 재계산).
  • 단일 store 멘탈모델 — Jotai로 atom 묶어 충분히 재현. 순수 취향.
  • React 밖 접근 — Jotai v2 getDefaultStore()·store.sub()로 가능(Provider 없을 때). Zustand가 유리한 건 "Provider 격리 store + 밖에서 접근"이라는 드문 교집합뿐.
→ Jotai가 못 하고 Zustand만 되는 근본적 갭은 사실상 없다. 둘은 ~90% 겹치는 같은 도구의 다른 모양.
✅ 권장 아키텍처 서버 상태(목록·캐싱·재검증)는 TanStack Query, 순수 클라 상태(모달·필터·선택)는 Zustand 또는 Jotai 하나. 둘을 섞지 말고 경계를 분명히(서버 데이터를 store에 복사하면 동기화 지옥).
→ 겹침이 ~90%라 결정은 능력이 아니라 ① 기존 코드베이스 ② 팀 멘탈모델 ③ 실제 쓰는 생태계 기능으로 한다. 단일 store 느낌·외부접근 기본 = Zustand · 파생·async(Suspense)·SSR 가벼움·atomFamily = Jotai.

Q. 우리 프로젝트는 jotai를 쓰는데, zustand로 갈아탈 이유가 있나? (앞 인사이트의 일반론을 우리 코드에 적용)
🏠 결론: jotai 유지, 마이그레이션 비권장. 이미 jotai의 차별 기능(atomWithStorage·atomFamily·useHydrateAtoms)을 핵심으로 쓰고, 멀티앱 Next.js + per-atom SSR 구조에 정확히 맞으며, 서버 상태는 TanStack Query가 분리 담당해 jotai 역할이 "UI/폼 상태"로 명확하다. zustand의 실익(응집도·React 밖 접근)은 현재 페인포인트가 아니다.

1. 사용 현황 (실측)

  • 버전 — 루트 중앙 관리: jotai@^2.1.0, jotai-family@^1.0.2
  • 규모 — 약 200개 atom (global-state 45 + 18개 도메인 모듈 154)
  • 구조 — @zaritalk/global-state가 useAlertState·useToastState·useCookieState 등 커스텀 훅으로 atom을 캡슐화. 도메인별 src/state/는 각자 로컬 전역상태
  • 활용 API — atomWithStorage(localStorage), atomFamily(쿠키 동적 키), useHydrateAtoms(SSR), atomWithReset
  • 역할 분담 — 서버 상태 = TanStack Query(약 1,600 사용처) · jotai = UI·폼·필터 전담
  • zustand — 현재 0건

3. 판단 — jotai 유지 (마이그레이션 비권장)

✅ 유지 근거

  • 차별 기능을 이미 핵심 사용 — atomWithStorage·atomFamily(쿠키)·useHydrateAtoms(SSR)는 zustand에선 미들웨어/수동 구현 → 단순 교체가 아니라 재설계.
  • 멀티앱 모노레포 + Next.js와 정합 — lessee/lessor/realtor/admin 등 6개 앱이 JotaiProvider로 스코프 격리 + atom 단위 SSR 주입. (apps/realtor/server/serverFetch.ts 같은 SSR 경로와 연결)
  • 비용 > 이득 — 200 atom + 커스텀 훅 + 18 도메인 교체 비용은 큰데, zustand 실익(응집도·React 밖 접근)은 현 페인포인트가 아님. 서버 상태는 TanStack Query가 책임져 jotai 역할이 "UI/폼"으로 명확 — jotai가 가장 잘하는 영역.
  • 훅 캡슐화가 잘 됨 — 앱 코드가 atom을 직접 import 안 하고 useAlertState()류만 사용 → 추상화 경계가 깔끔. 라이브러리 무관하게 좋은 설계라 바꿀 동기 약함.

4. zustand를 고려할 신호 (지금은 해당 없음)

📡 누적되면 일부 도메인에 부분 도입 검토

  • 한 도메인에서 상태+복잡한 액션 로직이 여러 atom에 흩어져 관리가 어려워질 때 (store action으로 응집)
  • React 밖(유틸·이벤트 핸들러)에서 전역상태를 직접 읽고/쓰는 요구가 빈번해질 때
  • atom 간 의존 그래프가 복잡해 디버깅이 힘들어질 때
⚠️ 그때도 전면 교체 X jotai 유지 + 특정 도메인만 zustand 병행이 현실적. 두 라이브러리는 공존 가능.
⚖️ 판단의 근거 = 능력이 아니라 적합성 jotai/zustand는 능력으로는 거의 무승부(응집도·단일 store·외부접근 모두 jotai로 재현 가능, 근본 갭 없음 — 인사이트 카드 참고). 그래서 선택은 ① 기존 코드베이스 ② 팀 멘탈모델 ③ 실제 쓰는 생태계 기능으로 갈리는데, 셋 다 jotai를 가리킨다(이미 200 atom · atomFamily·atomWithStorage·useHydrateAtoms 정착). "zustand가 불편해 보임"은 라이브러리 단점이 아니라 이 프로젝트의 적합 라이브러리가 jotai라는 신호.
✅ 최종 결론 현 시점 이 프로젝트(멀티앱 Next.js + 서버상태는 TanStack Query 분리 + 영속·SSR·동적키 활용)엔 jotai가 더 적합하며 마이그레이션은 권장하지 않는다. 앞 인사이트의 일반 규칙("값이 어디 사는가")에 비춰도, 우리는 atom=키 / 값은 Provider store에 격리 모델의 이점을 정확히 누리고 있다.
→ zustand 학습의 값어치는 마이그레이션이 아니라 남의 코드 읽기 + 라이브러리 무관 원리(persist·하이드레이션·지연초기화·경계 설계)를 jotai에 전이하는 것.

병렬·가로채기 라우팅 모달 — 도입 판단

📅 2026-06-21 실측라우팅 모달useNextQueryState의사결정
Q. 20강의 병렬(@slot)·가로채기((.)folder) 라우팅 모달을 우리 프로젝트에 도입할 가치가 있나?
🪟 결론: 도입 불필요. 두 패턴의 본질 가치는 딱 하나 — "모달 상태를 URL로 만든다". 그런데 이 프로젝트는 이미 useNextQueryState로 URL 연동 모달을 광범위하게 구현해 그 4대 이점을 다 누리고 있다. 갈아타면 얻을 게 거의 없고 Pages Router 전면 마이그레이션 + 네이티브 앱스택 히스토리 충돌 비용만 든다.

1. 두 패턴의 본질 가치 = "모달을 URL로"

병렬·가로채기 라우팅 모달이 주는 건 결국 4가지로 수렴:

  • 공유·북마크 가능한 모달 — 링크 보내면 상대도 모달 열린 채로 봄(사진/게시물 모달)
  • 뒤로가기로 자연스럽게 닫힘 — 수동 처리 제거
  • 새로고침 폴백 — 가로채기 라우트는 닫히고 진짜 페이지로
  • SEO/딥링크 — 모달 콘텐츠가 실제 라우트
→ "콘텐츠가 그 자체로 URL을 가질 가치가 있는 모달"(상품 상세·게시물·사진)일 때만 빛난다.

2. 이미 갖춰진 것 — useNextQueryState

핵심 훅은 libs/shared/.../useNextQueryState.ts. 패턴:

// 열기
router.push(appendQueryParamsToPath(asPath, { key: 'true' }), { shallow: true });
// 읽기
const [isOpen] = useNextQueryState('key', { parse, defaultValue });
// 닫기
router.back();  // 또는 deleteQueryState()
병렬/가로채기 라우팅 이점useNextQueryState로 이미 달성
공유·북마크 가능한 모달 URL✅ ?reviewImageZoom=true 등
뒤로가기로 닫힘✅ router.back()(history 기반)
새로고침 시 상태 유지✅ URL 쿼리에서 다시 읽음
딥링크로 모달 열린 채 진입✅ airbridge router의 modalTarget 등

3. 가로채기 라우팅만의 유일한 실질 차이

"같은 URL을 진입 맥락에 따라 모달로도 / 풀페이지로도" — (.)로 클릭하면 모달, 그 URL을 새로고침·직접 진입하면 동일 콘텐츠가 풀페이지로.

현 useNextQueryState는 ?imageZoom=true 새로고침 시 풀페이지가 아니라 피드 위 모달로 복원. 대부분은 이게 자연스럽고, 풀페이지 폴백이 필요하면 이미 pushWithAppStack으로 별도 라우트를 push하면 됨.

→ 가로채기 라우팅 = "쿼리 모달 + 앱스택 라우트 push" 두 도구로 이미 커버.

4. 결정적 제약 3가지

❌ 도입 시 충돌

  • 주력 앱이 Pages Router — 모달 많은 핵심 앱(lessee/lessor)이 Pages Router → App Router 전면 마이그레이션 선행(모달 때문에 감수할 비용 아님).
  • 네이티브 앱스택과 이중 관리 — 웹뷰에서 pushWithAppStack/back으로 앱스택·히스토리 동기화 중. 가로채기 라우팅의 자체 히스토리 조작이 끼면 "뒤로가기가 모달을 닫나 화면을 닫나" 충돌(changeUrlAfterStackPop 등과 부딪힘).
  • 전역 모달과 모델 불일치 — alert/toast/confirm은 어디서든 명령형 호출(setAlertState({...}) 한 줄). 라우트로 바꾸면 라우트 정의 + slot + default.tsx로 무거워짐.

5. 권장 — 도입 대신 현 패턴 정리 (ROI ↑)

✅ 다듬을 것

  • 열기/닫기 일관성 — push + router.back()와 replace + deleteQueryState() 혼재. replace로 연 모달은 history에 안 쌓여 뒤로가기로 안 닫힘. "뒤로가기로 닫을 모달=push / URL 정리용 일시상태=replace" 규칙 명문화.
  • boolean 쿼리 모달 래퍼 — isReservationPaymentCompletePopupOpen, isOpenHidingVacancy… 처럼 === 'true' 반복 → useBooleanQueryModal(key) 얇은 래퍼로 보일러플레이트 축소(미래 "쿼리=가로채기" 표준 진입점).
✅ 최종 결론 판단 기준은 "모달을 URL로 만들 가치가 있나" 하나인데, 이 프로젝트는 이미 그렇게 하고 있고(useNextQueryState) 잘 하고 있다. 병렬/가로채기로 갈아타면 얻는 것 거의 없고(본질 가치 이미 확보), Pages 마이그레이션·앱스택 충돌·명령형 전역모달 무거워짐만 떠안는다. → 현 패턴 유지 + 정리가 정답. (jotai 판단과 동일 결: 도입이 아니라 적합성)
