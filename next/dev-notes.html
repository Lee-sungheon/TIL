SSR과 전역 상태 격리

📅 2026-06-20 ReactSSR React QueryJotai
Q. QueryClient를 useState로 감싸는 이유는? 'use client'인데 어떻게 다른 유저와 섞이지? 모듈 밖 선언은 다 위험한가?
💡 서버는 하나의 프로세스로 여러 요청을 동시에 처리한다. 그래서 모듈 레벨(함수 밖) 변수는 요청 간 공유된다 — 가변 상태라면 이게 사용자 간 데이터 누출이 된다.

1. 모듈 레벨 선언은 서버에서 공유된다 (항상 위험한 건 아님)

판별 기준은 딱 하나: "요청·사용자마다 달라져야 하는 가변 상태인가?"

✅ 공유돼도 안전

  • 상수 const MAX = 3
  • 순수 함수 formatDate()
  • 컴포넌트 정의 function Button()
  • Jotai atom(0) — 정의(키)일 뿐

❌ 공유되면 위험

  • new QueryClient() — 캐시 보유
  • let currentUser — 요청별 변동
  • const cache = new Map()
  • 요청 데이터를 담는 모든 가변 전역
🏭
공장(서버)에 설명서·공구(상수·함수)는 같이 써도 되지만, 주문서·장바구니(가변 상태)는 손님마다 따로 있어야 한다.

2. 'use client' ≠ "서버에서 안 돈다"

'use client'는 "클라이언트 컴포넌트 경계" 표시일 뿐. 클라이언트 컴포넌트도 첫 요청 때는 서버에서 한 번 렌더링(SSR)된 뒤 브라우저에서 hydration된다. 그래서 모듈 레벨 가변 상태는 이 서버 SSR 단계에서 섞인다.

참고 브라우저는 한 사용자 = 한 탭 = 독립 환경이라 애초에 섞일 일이 없다. 문제는 항상 서버 측.

3. React Query 패턴

QueryClient는 정의 + 캐시(상태)가 한 덩어리 → 인스턴스째 격리해야 한다.

// 매 렌더 새로 생성 방지 + 요청별 격리 동시 해결
const [queryClient] = useState(() => new QueryClient({...}));
방식리렌더 재생성 방지요청 간 격리
함수 안 직접 선언✗—
모듈 레벨✓✗
useState✓✓

4. Jotai 패턴

핵심은 정의(atom)와 값(store)을 분리하는 것.

  • atom = 값이 아니라 키(정의) → 모듈 전역에 둬도 안전
  • 값은 store에 저장, 격리는 <Provider>로 (요청마다 새 store)
  • useHydrateAtoms로 서버 초기값 주입 → hydration mismatch 방지
'use client';
<Provider>{children}</Provider>            // store 격리
useHydrateAtoms([[dataAtom, initialData]]); // 서버 값 주입
⚠️ 주의 Provider 없이 쓰면 전역 default store 사용 → React Query와 같은 누출 위험.
✅ 최종 판단 기준
모두에게 똑같고 안 바뀌면 → 밖에 둬도 OK. 사용자마다 달라야 하면 → useState나 Provider로 격리.
· React Query: 인스턴스째 격리(정의 = 상태) · Jotai: 정의는 전역, store만 격리(정의 ↔ 상태 분리)

isServer → environmentManager : 환경 판정의 진화

📅 2026-06-20 React QuerySSR Next.js런타임 감지
Q. 강의(11강)의 import { isServer } from '@tanstack/react-query'가 deprecated되고 environmentManager.isServer()로 바뀌었다. 왜? 의도가 뭔가?
💡 isServer는 모듈 로드 시 1회 박제되는 상수였다. 그 단순 판정(typeof window === 'undefined')이 크롬·VSCode 확장의 service worker와 edge 런타임에서 틀려 refetch가 죽는 실제 버그가 있었고, 이를 호출 시점에 평가 + 전역 override 가능한 함수로 바꾼 게 environmentManager다.

1. 무엇이 바뀌었나 — 상수 → 함수(+manager)

기존 isServerenvironmentManager.isServer()
형태상수 (boolean)메서드 (함수 호출)
평가 시점import 시 1회 박제호출할 때마다
override✗ 불가✓ setIsServer(fn)
현재 상태deprecated권장

판정 로직 자체는 동일(typeof window === 'undefined' || 'Deno' in globalThis)하다. 달라진 건 "상수로 굳히느냐, 함수로 매번 묻느냐"뿐.

2. 강의 코드에서 이게 하는 일 = SSR 상태 격리

import { environmentManager } from '@tanstack/react-query';

function getQueryClient() {
  if (environmentManager.isServer()) {   // 호출 시점에 평가
    return makeQueryClient();            // 서버: 매 요청 새 인스턴스(격리)
  }
  if (!browserQueryClient) browserQueryClient = makeQueryClient();
  return browserQueryClient;              // 브라우저: 싱글턴 재사용
}

이 분기의 정체는 ENTRY #1의 SSR 상태 격리 그 자체다 — 서버면 요청마다 새 QueryClient(캐시 누출 방지), 브라우저면 싱글턴 재사용. 즉 환경 판정이 격리 전략을 가른다.

3. 왜 바꿨나 — 진짜 동기는 "실제 버그"

❌ 상수가 오판하던 환경 (issue #10139 · #5575)

  • 크롬·VSCode 확장의 background service worker: window가 없다. 하지만 여긴 서버가 아니라 클라이언트다 → 상수가 isServer = true로 오판 → focus/online refetch 등 클라 전용 로직이 전부 꺼짐 → "refetch가 안 돼요" 버그
  • Edge 런타임(Cloudflare 등): 전통적 server/browser 이분법에 안 맞아 동일하게 오작동

상수였기 때문에 생긴 두 가지 근본 한계 → manager로 해소:

상수의 한계

  • 고칠 수 없다 — import 시점에 boolean으로 굳어 끼어들 틈이 없음
  • 타이밍 고정 — 런타임 변화·테스트 교체 불가

manager의 이점

  • override 가능 — 커스텀 판정 주입
  • call-time 평가 — 동적 환경·테스트에 유연
  • 중앙화 — 흩어진 체크를 단일 싱글턴으로

4. override 방법

// "이 런타임은 클라이언트로 취급하라"를 전역 주입
environmentManager.setIsServer(
  () => typeof window === 'undefined' && !('chrome' in globalThis)
);
마이그레이션 강의 코드는 당장 안 고쳐도 동작한다(기본 판정 로직 동일, deprecation 경고만 뜸). 새로 짠다면 environmentManager.isServer()를 쓰면 된다.
📌 핵심
· 표면: 상수 isServer → 함수 environmentManager.isServer() (기본 동작 동일)
· 의도: 모듈-로드-시 박제되는 상수로는 확장 worker·edge 런타임의 오판을 고칠 수 없어서, 호출 시점 평가 + setIsServer override를 연 것
· 연결: 이 판정이 곧 SSR 상태 격리의 분기점 — 환경을 잘못 읽으면 격리도 깨진다

getQueryClient의 정석 — cache vs isServer, 그리고 결합

📅 2026-06-20 React QuerySSR Next.js런타임 감지
Q. 강의에서 cache(() => new QueryClient())(서버)와 isServer 분기(클라 Provider)가 혼재한다. 상황별로 쓰는 건가, 통일하는 건가? 실전 베스트 셋업은?
💡 공식 권장은 isServer(→environmentManager.isServer()) 분기 하나로 통일이다(cache는 대안). 가장 견고한 형태는 서버 분기를 cache로 감싼 결합형 — 서버는 요청 스코프 싱글턴, 브라우저는 모듈 싱글턴, 진입점은 isServer로 통일.

1. 강의는 두 맥락에서 두 패턴을 섞어 쓴다

강패턴맥락
11강 providers.tsxisServer 분기 + 모듈 싱글턴클라이언트 Provider
12·14강 page.tsxcache(() => new QueryClient())서버 컴포넌트 prefetch

둘 다 목표는 같다 — "서버는 요청 간 격리, 브라우저는 하나 재사용". 각각은 동작하지만 통일은 안 돼 있다.

2. 공식 권장 = "하나로 통일", 그 하나는 isServer 분기

TanStack Query 공식 SSR 가이드는 isServer 분기 패턴 하나를 app/get-query-client.ts에 두고 서버 컴포넌트와 클라 Provider가 같은 함수를 import하게 한다. cache()는 "alternative approach"로만 언급된다.

왜 cache는 메인이 못 되나 cache()는 React Server Component 전용 기능이라 'use client' Provider와 공유 불가 → "서버용 / 클라용" 둘로 쪼개진다. 반면 isServer 분기는 한 함수가 서버/브라우저를 다 처리 → 단일 함수로 통일된다.

3. ⭐ 베스트 = 둘을 결합 (cache + isServer)

공식 메인(isServer)의 유일한 미세 약점은 서버 분기가 return makeQueryClient()라 호출마다 새 인스턴스인 것. 이걸 cache로 감싸면 약점이 사라진다.

import { QueryClient, QueryCache, environmentManager } from '@tanstack/react-query';
import { cache } from 'react';

function makeQueryClient() {
  return new QueryClient({ /* defaultOptions, queryCache 등 */ });
}

const getServerQueryClient = cache(makeQueryClient);   // ① 서버: 요청 스코프 싱글턴
let browserQueryClient;                                 // ③ 브라우저: 모듈 싱글턴

export function getQueryClient() {
  if (environmentManager.isServer()) {
    return getServerQueryClient();                      // ② 진입점 통일
  }
  if (!browserQueryClient) browserQueryClient = makeQueryClient();
  return browserQueryClient;
}
  • ① cache(makeQueryClient) — 서버에서 요청 단위로 메모이즈. 같은 요청 내 여러 서버 컴포넌트가 호출해도 동일 인스턴스(prefetch가 한 캐시에 모임), 다른 요청끼린 격리.
  • ② environmentManager.isServer() — 한 함수가 서버/브라우저를 다 처리 → 서버·클라 공유.
  • ③ 브라우저 모듈 싱글턴 — Suspense로 초기 렌더가 끊겨도 client 재생성 방지.
항목isServer만
(공식 메인)		cache만
(12강)		⭐ 결합형
서버·클라 한 함수 통일✓✗ 서버전용✓
서버 요청 내 동일 인스턴스✗ 매번 새로✓✓
브라우저 싱글턴(Suspense 안전)✓✗✓
요청 간 격리✓✓✓

→ 결합형은 모든 칸이 ✓. 공식 패턴보다 한 단계 더 견고하다.

📌 결론
· 통일이 정답: 상황별이 아니라 하나의 getQueryClient를 서버·클라가 공유 (공식 = isServer 분기)
· 왜 isServer가 축: cache는 RSC 전용이라 단독으론 클라와 공유 불가 — 통일의 열쇠는 isServer
· 베스트: 서버 분기를 cache로 감싼 결합형 = 서버 요청스코프 싱글턴 + 브라우저 모듈 싱글턴 + 진입점 통일
· 연결: ENTRY #2의 environmentManager가 바로 이 분기의 판정자

Suspense ≠ hydration 방어막 & initialDataUpdatedAt

📅 2026-06-20 Suspensehydration Next.jsReact Query
Q. 강의는 "Suspense가 클라 컴포넌트의 Date.now() hydration 에러를 막는 방어막"이라는데 맞나? Suspense를 감싸면 getCachedData(서버 캐시)에 영향을 받나?
💡 실측 결론(§6): Suspense는 없애면 안 된다 — 빌드가 실패한다. 단 이유는 강의 설명과 다름. hydration mismatch가 아니라, TanStack Query가 staleTime 계산에 Date.now()를 쓰는데 Next.js 16 "use cache" 모드가 Suspense 없는 현재시각 접근을 금지하기 때문. 순수 hydration만 보면 시간을 화면에 출력할 때만 mismatch가 난다.

1. getCachedData는 Suspense 바깥에서 끝난다

const { posts, serverTime } = await getCachedData(); // ← JSX 만들기 전에 완료 (Suspense 밖)
return (
  <Suspense fallback={...}>          // ← await가 끝난 "뒤"에 생성되는 JSX
    <PostListInitial ... />
  </Suspense>
);

await는 JSX를 만들기 전 함수 본문 맨 위에서 실행된다. 데이터를 다 받은 뒤에 Suspense가 포함된 JSX가 생긴다 → 캐시 로딩과 시점이 겹치지 않음 → 무관.

2. PostListInitial은 애초에 suspend하지 않는다

Suspense는 자식이 suspend할 때만 fallback을 띄운다. 그런데:

  • 일반 useQuery 사용 (useSuspenseQuery 아님) → 절대 suspend 안 함
  • initialData까지 있어 첫 렌더부터 데이터 완비 → 기다릴 일이 없음

→ fallback은 영원히 안 뜬다(react-query 관점에서 Suspense는 할 일이 없음). 단, 이것만 보고 "그러니 Suspense 불필요"로 결론내면 틀린다 — 별개의 이유로 필수다. 아래 §6 실측 참고.

3. 그래서 hydration mismatch가 "없다"

강의 설명이 맞으려면 클라 컴포넌트가 렌더 중 new Date()를 직접 출력해야 한다. 하지만 실제 코드의 Date.now()는 서버 함수 안에 있고, 한 번 계산되어 직렬화된 props(updatedAt)로 내려온다.

시간 계산 위치서버/클라 값mismatch
강의가 가정한 코드클라 컴포넌트 렌더 중다름 (10초 vs 11초)✅ 발생
실제 이 코드서버 함수 안 (1회) → props동일 (같은 숫자)❌ 불가능

게다가 updatedAt은 화면에 출력조차 안 됨(useQuery 설정값으로만 쓰임) → mismatch 날 텍스트 자체가 없다.

4. Suspense는 "예방"이 아니라 "격리"

⚠️ 메커니즘 교정 mismatch는 Suspense가 있어도 발생한다(서버가 값을 박고 클라가 다른 값으로 hydrate). Suspense는 그때 경계 안쪽만 버리고 클라에서 다시 그려 복구할 뿐 — 폭발 반경을 가두는 에러 격리지 예방이 아니다. 진짜 예방책은 useEffect(마운트 후 계산)·suppressHydrationWarning·클라 전용 렌더.

5. initialDataUpdatedAt의 진짜 역할

"안 주면 즉시 refetch한다"는 staleTime이 기본값(0)일 때 얘기다. 이 코드는 staleTime: 5분이라 안 줘도 마운트 직후 refetch는 안 한다.

진짜 가치는 "use cache"로 묵은 캐시의 실제 나이를 반영하는 것 — staleTime 카운트의 기준 시각을 서버 제조 시각으로 이동.

"use cache"로 getCachedData 결과가 4분 전에 캐시됨 → serverTime = "4분 전" 값

✅ 줌  : 기준 4분 전 + staleTime 5분 → 1분 뒤 stale (정확)
❌ 안줌: 기준 "지금" + staleTime 5분 → 5분 뒤 stale
        → 이미 4분 묵은 데이터를 총 9분간 fresh로 오판

강의 프로젝트를 실제로 빌드·실행해 여러 케이스를 측정했다 (puppeteer로 hydration 콘솔 수집 + next build).

→ 셋의 교집합일 때만 실패한다: "use cache" 모드 × react-query의 Date.now() × Suspense 없음. 하나만 빠져도 통과(A·E·G가 증거).

단서 이 강제는 "use cache"/cacheComponents를 켰을 때만이다. 이 모드가 아니면 react-query의 Date.now()는 무해하고 Suspense도 불필요했다. 즉 Suspense를 부른 건 hydration이 아니라 "서버 캐시 모드 + 클라이언트의 동적 시각 접근"의 조합 — 바로 이 강의의 주제다.
📌 강의 주장 vs 실측
· "Suspense가 필요하다" → ✅ 결론은 맞음 (없애면 빌드 실패)
· "이유 = 클라 Date.now() hydration mismatch 방어" → ❌ 부정확. 시간을 화면에 출력할 때만 mismatch(C). B는 mismatch 없이도 실패
· 진짜 이유 → react-query가 staleTime에 Date.now() 사용 + "use cache" 모드의 Suspense-less 현재시각 금지
· "initialDataUpdatedAt 중요" → ✅ 맞음 (묵은 캐시의 실제 나이 반영)

dehydrate(queryClient)의 원리 & staleTime은 어디에 두나

📅 2026-06-20 React Queryhydration SSRprefetch
Q. dehydrate(queryClient)는 무슨 원리로 동작하나? 그리고 강의의 "서버 getQueryClient의 staleTime이 브라우저 재요청을 막는다"가 맞나? staleTime이 지나면 데이터는 버려지나?
💡 dehydrate는 캐시에서 데이터 + 받은 시각(dataUpdatedAt)만 뽑아 JSON으로 만든다 — staleTime 같은 설정은 안 뽑는다. 그래서 ① 클라 refetch를 막는 staleTime은 반드시 클라이언트 QueryClient에 둬야 하고(서버 것은 안 건너감), ② staleTime이 지나도 데이터는 버려지지 않는다 — 버리는 건 gcTime.

1. dehydrate의 원리 — "캐시의 데이터만 진공포장"

왜 필요한가: 서버 컴포넌트 → 클라 컴포넌트로 넘기려면 값이 직렬화(JSON) 가능해야 한다. 그런데 QueryClient 인스턴스(클래스·메서드·내부 캐시 구조)는 직렬화 불가 → 그대로 못 넘긴다. 그래서 캐시에서 데이터 부분만 순수 객체로 추출하는 게 dehydrate다.

무엇을 추출하나 (소스 기준): queryCache.getAll()을 순회하며 성공한(success) 쿼리만 골라 아래 형태로 만든다.

dehydrate(queryClient) ──▶ {
  queries: [{
    queryKey, queryHash,
    state: { data, dataUpdatedAt },   // ← 데이터 + "받은 시각"만
  }],
  mutations: [ ... ],
}
// ❌ staleTime · gcTime · queryFn 등 "설정"은 직렬화되지 않는다

클라이언트는 이 객체를 hydrate()로 받아 queryCache.build()로 자기 캐시에 데이터를 심는다. 이때 적용되는 staleTime 등 설정은 클라이언트 자신의 defaultOptions이지, dehydrate된 값이 아니다.

🥫
서버는 음식에 제조일자(dataUpdatedAt)만 찍어 진공포장해 보낸다. 유통기한(staleTime)을 며칠로 볼지는 받는 쪽 냉장고(클라이언트)가 자기 규칙으로 정한다.

2. 그래서 — 클라 refetch를 막는 staleTime은 "클라이언트"에 둬야 한다

강의는 서버 getQueryClient에만 staleTime: 60초를 주고 "이게 브라우저 재주문을 막는다"고 설명한다. 하지만 위 원리상 서버의 staleTime은 dehydrate에 안 실려 클라로 건너가지 않는다. 클라는 hydrate된 dataUpdatedAt을 자기 자신의 staleTime과 비교해 stale 여부를 판단한다.

staleTime을 둔 위치실제 영향 범위클라 마운트 refetch 차단?
서버 getQueryClient (강의)서버 prefetch 단계의 신선도만✗ (dehydrate에 안 실림)
클라이언트 QueryClienthydrate된 데이터의 신선도 판정✓
⚠️ 함정 강의의 클라 PostList의 useQuery엔 staleTime이 없다. 이 시리즈는 11강 Providers의 makeQueryClient가 staleTime: 60초라 우연히 동작할 뿐 — "서버 매니저 staleTime이 브라우저 재요청을 막는다"는 인과는 틀렸다. 클라 staleTime이 0이면 prefetch해도 마운트 즉시 background refetch가 난다.

3. staleTime이 지나도 데이터는 "버려지지 않는다"

강의는 "유통기한 0초 → 완성되자마자 상했다고 판정"이라 비유하는데, stale = 폐기가 아니다. stale이 돼도 데이터는 캐시에 남아 즉시 화면에 계속 제공되고, 다음 트리거(마운트·포커스·재연결) 때 뒤에서 새로 받아온다(stale-while-revalidate). 실제로 메모리에서 삭제하는 건 gcTime이다.

staleTimegcTime
의미"신선함" 기한"폐기" 기한
지나면stale 표시 — 데이터 유지·계속 표시, 재요청 후보메모리에서 제거

교정 "상했다 = 버린다"는 staleTime을 gcTime과 혼동시킨다. 정확히는 "신선표시만 떼고 그대로 쓰되, 다음 기회에 새로 받아온다"가 staleTime 경과의 의미다.
📌 핵심
· dehydrate 원리: 캐시에서 데이터 + dataUpdatedAt만 진공포장(설정은 제외) → 직렬화해 클라로 전달
· 귀결①: 신선도 판정(staleTime)은 받는 쪽(클라이언트) 책임 — 서버에 줘봐야 클라 refetch는 못 막는다
· 귀결②: staleTime 경과 = "다시 받을 후보" 표시일 뿐, 폐기가 아니다(폐기는 gcTime)
· 연결: ENTRY #4의 initialDataUpdatedAt도 같은 축 — "데이터가 언제 만들어졌나"라는 기준 시각을 정확히 넘기는 문제

initialData vs dehydrate — 두 SSR 데이터 전달 방식

📅 2026-06-20 React Queryhydration SSRprefetch
Q. 서버 컴포넌트에서 데이터를 미리 패칭해 initialData로 내려주는 것과, prefetchQuery 후 dehydrate하는 것의 차이는?
💡 initialData는 데이터를 props로 컴포넌트에 직접 쥐여주는 것, dehydrate는 서버 캐시 스냅샷을 통째로 넘겨 경계 아래가 알아서 꺼내 쓰게 하는 것. 후자가 신선도·확장성·정확성에서 우위라 공식 권장이다.

1. 본질 — "직접 주입" vs "캐시째 직렬화"

// A) initialData — props로 컴포넌트에 직접 주입
const posts = await getData();
<PostListInitial initialData={posts} updatedAt={serverTime} />   // 컴포넌트를 알아야
//   클라: useQuery({ queryKey:['posts'], initialData: posts, initialDataUpdatedAt: updatedAt })

// B) dehydrate — 캐시를 직렬화해 경계 아래 어디서나
const qc = getQueryClient();
await qc.prefetchQuery({ queryKey:['posts'], queryFn });
<HydrationBoundary state={dehydrate(qc)}>
  <PostList />        // 그냥 useQuery({ queryKey:['posts'] }) — initialData 불필요
</HydrationBoundary>

A는 데이터와 컴포넌트가 강결합(그 컴포넌트·queryKey를 알아야 함), B는 분리(경계 아래 같은 queryKey면 어디서든 접근).

2. 공식이 명시한 initialData의 caveat

관점A) initialDataB) dehydrate

신선도 기준 시각"페이지 로드 시점" → initialDataUpdatedAt 수동서버 실제 fetch 시각 자동 보존
이미 캐시에 데이터절대 안 덮어씀(더 신선해도)갱신 가능

3. 우리 노트와 직결되는 두 함정

① 신선도 기준 시각 initialData는 dataUpdatedAt을 "지금(hydration 시점)"으로 잡는다. "use cache"로 4분 묵은 데이터를 줘도 클라는 "방금 받음"으로 착각 → 그래서 10강이 updatedAt={serverTime}(=initialDataUpdatedAt)을 수동으로 넘긴 것. ENTRY #5에서 봤듯 dehydrate는 이 시각을 자동 직렬화해 수동 작업이 없다.
② stale 덮어쓰기 안 함 initialData는 해당 queryKey 캐시가 비어있을 때만 초기값으로 들어가고, 이미 데이터가 있으면 더 신선해도 무시한다. dehydrate엔 이 제약이 없다.

4. 언제 무엇을

📦 dehydrate (공식 권장 · 메인)

  • 여러 쿼리 · 중첩 · 깊은 트리
  • 정확한 신선도(dataUpdatedAt)가 중요
  • status/error까지 전달

⚡ initialData (빠른 케이스)

  • 단일 쿼리 + 얕은 트리
  • 셋업 최소화가 우선
  • 신선도 정밀도가 덜 중요
강의 매핑 10강 = initialData 방식(PostListInitial) · 12강 = dehydrate 방식(PostList). 강의가 두 패턴을 발전 순서로 보여준 것.
📌 핵심
· 본질: initialData = 컴포넌트에 직접 쥐여주기 / dehydrate = 캐시 스냅샷 통째 넘기고 경계 아래가 꺼내 씀
· 공식 권장: dehydrate(Hydration) — 신선도·확장성·정확성 우위, initialData는 단일·얕은 트리의 빠른 해결책
· 연결: ENTRY #4(initialDataUpdatedAt) ↔ ENTRY #5(dehydrate)를 잇는 다리

무한 쿼리의 함정 — maxPages vs 가상화(virtualization)

📅 2026-06-20 React Query무한스크롤 가상화성능
Q. 14강 무한 스크롤(useInfiniteQuery) 코드에 잘못된 정보는? 그리고 maxPages는 정확히 뭘 하고, 가상화(virtualization)와는 뭐가 다른가?
💡 강의가 강조한 maxPages는 ① 실제 코드엔 주석으로 꺼져 있고 ② 켜도 getPreviousPageParam이 없어 설명대로 동작 안 한다. 그리고 핵심: maxPages는 "캐시 데이터(JS 힙)"를, 가상화는 "DOM 노드"를 줄인다 — 14강이 노린 "수천 이미지 DOM 폭발"은 사실 가상화의 영역이다.

1. 강의 검증 — 붙여넣은 설명 ≠ 실제 배포 소스

항목강의 설명실제 소스
maxPages: 4활성(길게 강조)// maxPages: 4 주석(비활성)
initialPageParam서버 1 / 클라 1서버 1 / 클라 2 불일치
page.tsx Suspense없음있음(GalleryPrefetch 분리)
getPreviousPageParam없음없음

❌ 핵심 오류 3가지

  • A. maxPages: 실제 코드엔 주석(비활성) → 테스트 4·5번("위로 올리면 채워진다", "지우면 메모리 누수") 재현 불가. 켜도 getPreviousPageParam·fetchPreviousPage가 없어 제거된 페이지 복구 불가. 제거된 페이지는 data.pages→DOM에서 완전 삭제(빈 뼈대로 남는 건 가상화/lazy-load 동작). "양방향"이라지만 코드는 단방향.
  • B. initialPageParam: 서버 prefetch는 1, 클라는 2 → 같은 queryKey인데 시작점 불일치. 캐시 미스·refetch 시 page 1 누락 가능.
  • C. maxPages ↔ 스크롤 보정 충돌: maxPages를 켜면 pages.length가 4에서 고정 → "페이지 증가 감지" useEffect가 5페이지째부터 4>4=false → scrollBy 멈춤.
✅ 정확한 부분 staleTime을 클라 useInfiniteQuery에 직접 설정(ENTRY #5 기준 올바름, 12강보다 개선) · async prefetch를 <Suspense>로 감쌈 · flatMap 평탄화 · getNextPageParam v5 시그니처.

2. maxPages 심화 — "Limited Infinite Query"

한쪽 방향으로 새 페이지를 가져오면 반대쪽 끝 페이지가 캐시에서 제거되어, data.pages 배열이 항상 최대 N개로 유지된다.

maxPages: 4
pages: [P1, P2, P3, P4]
fetchNextPage()  ─▶  pages: [P1 제거, P2, P3, P4, P5] → [P2, P3, P4, P5]
                       // 위로 돌아가려면 getPreviousPageParam + fetchPreviousPage 필요

공식이 밝힌 두 가지 동기:

① 메모리

페이지가 수십~수백 개 쌓이는 걸 방지(JS 힙 데이터).

② refetch 비용

무한쿼리는 stale 시 모든 페이지를 순차 재fetch → maxPages면 보관분만 refetch.

⚠️ getNext + getPrevious 둘 다 필수 maxPages가 한쪽 끝을 버리므로, 반대로 스크롤하면 그 페이지를 다시 가져와야 한다. 그래서 양방향 파라미터가 모두 있어야 정상 동작한다(강의 코드엔 getNextPageParam만 있음 → 그래서 maxPages를 주석 처리한 듯).

3. maxPages vs 가상화 — "줄이는 대상"이 다르다

둘 다 "메모리 최적화"로 묶이지만 작동 층위가 완전히 다르다.

🍽️
maxPages = 주방 냉장고(데이터)에 재료를 4묶음만 보관(넘으면 버리고, 다시 필요하면 장보기=네트워크). 가상화 = 식탁(DOM)에 지금 먹을 접시만 올리고 나머지는 창고에 그대로(언제든 즉시 꺼냄=재요청 없음).

4. 그래서 14강의 도구 선택은?

강의는 "수천 장의 이미지 접시가 쌓여 메모리 폭발"을 maxPages로 막는다고 했다. 하지만 그건 DOM 노드 문제 → 가상화의 영역이다. maxPages는 데이터를 지워 대가로 네트워크 재요청을 치르는 도구라, DOM 성능의 정답이 아니다(게다가 주석이라 작동도 안 함).
